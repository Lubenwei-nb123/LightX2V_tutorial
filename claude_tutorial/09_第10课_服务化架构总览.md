### 1.1 Rank 0 与非 Rank 0 的分工

```
                    ┌──────────────────────────┐
                    │      torchrun 启动       │
                    │  N 个进程, 每个一张 GPU   │
                    └────────┬─────────────────┘
                             │
              ┌──────────────┴──────────────┐
              │                             │
       ┌──────┴──────┐              ┌──────┴──────┐
       │   Rank 0    │              │  Rank 1..N  │
       │             │              │             │
       │ FastAPI     │              │ worker_loop │
       │ 监听 HTTP   │  broadcast   │ 等待任务    │
       │ 请求        │ ──────────→  │ 数据        │
       │             │  (gloo)      │             │
       │ process_    │              │ process_    │
       │ request()   │  barrier     │ request()   │
       │             │ ←─────────→  │             │
       └─────────────┘              └─────────────┘
```

**关键设计**：只有 Rank 0 运行 FastAPI 和 TaskManager。非 Rank 0 进程通过 `DistributedManager.broadcast_task_data()` 从 Rank 0 接收任务数据（使用 gloo 后端 + pickle 序列化），然后各自在自己的 GPU 上执行推理。

## 5. ServiceContainer：依赖注入

**核心文件**：`lightx2v/server/api/deps.py`

```python
class ServiceContainer:
    _instance: Optional["ServiceContainer"] = None   # 单例

    def __init__(self):
        self.file_service = None
        self.inference_service = None
        self.video_service = None
        self.image_service = None
        self.max_queue_size = 10

    @classmethod
    def get_instance(cls) -> "ServiceContainer":
        if cls._instance is None:
            cls._instance = cls()
        return cls._instance

    def initialize(self, cache_dir, inference_service, max_queue_size=10):
        self.file_service = FileService(cache_dir)
        self.inference_service = inference_service
        self.video_service = VideoGenerationService(self.file_service, inference_service)
        self.image_service = ImageGenerationService(self.file_service, inference_service)
        self.max_queue_size = max_queue_size

def get_services() -> ServiceContainer:
    return ServiceContainer.get_instance()
```

**设计要点**：所有 API handler 通过 `get_services()` 获取服务实例，不直接持有引用。这样 `initialize()` 可以在任意时机调用，handler 注册时不需要服务已就绪。

## 6. TaskManager：任务队列与状态机

**核心文件**：`lightx2v/server/task_manager.py`

### 6.1 任务状态机

```
                create_task()
                     │
                     ▼
              ┌─────────────┐
              │   PENDING    │ ←── 任务创建，等待处理

                 │    │

          └──────┘  └────────┘  └───────────┘
```

### 6.2 TaskInfo 数据结构

@dataclass

### 6.3 并发控制：双锁机制

TaskManager 使用**两把锁**实现并发控制：

```python
class TaskManager:
    def __init__(self):
        self._lock = threading.RLock()            # 1. 队列锁（可重入）
        self._processing_lock = threading.Lock()  # 2. 处理锁（互斥）
        self._current_processing_task = None
```

| 锁 | 类型 | 保护的资源 | 使用场景 |
|----|------|-----------|---------|
| `_lock` | RLock（可重入） | `_tasks` OrderedDict | 所有队列读写操作 |
| `_processing_lock` | Lock（互斥） | GPU 推理资源 | 确保同一时刻只有一个任务在推理 |

**为什么用两把锁？**

`_lock` 保护内存中的队列数据结构，粒度小、持有时间短。`_processing_lock` 保护 GPU 推理过程，粒度大、持有时间长（一次推理可能几十秒）。如果用一把锁，查询任务状态的 API 会被正在推理的任务阻塞。

### 6.4 任务处理流程

```python
# server.py:46-65  _task_processing_loop（守护线程）
def _task_processing_loop(self):
    loop = asyncio.new_event_loop()        # 创建独立的事件循环

    while not self.stop_processing.is_set():
        task_id = task_manager.get_next_pending_task()  # FIFO 取第一个 PENDING 任务

        if task_id is None:
            time.sleep(1)                  # 无任务时轮询（1秒间隔）
            continue

        task_info = task_manager.get_task(task_id)
        if task_info and task_info.status == TaskStatus.PENDING:
            loop.run_until_complete(self._process_single_task(task_info))
```

```python
# server.py:67-112  _process_single_task
async def _process_single_task(self, task_info):
    task_id = task_info.task_id
    message = task_info.message

    # 1. 获取处理锁（保证单任务推理）
    lock_acquired = task_manager.acquire_processing_lock(task_id, timeout=1)
    if not lock_acquired:
        task_manager.fail_task(task_id, "Failed to acquire processing lock")
        return

    try:
        # 2. 状态迁移：PENDING → PROCESSING
        task_manager.start_task(task_id)

        # 3. 检查是否已被取消
        if task_info.stop_event.is_set():
            task_manager.fail_task(task_id, "Task cancelled")
            return

        # 4. 根据请求类型选择生成服务
        if isinstance(message, ImageTaskRequest):
            generation_service = services.image_service
        else:
            generation_service = services.video_service

        # 5. 执行生成（核心推理在这里发生）
        result = await generation_service.generate_with_stop_event(
            message, task_info.stop_event
        )

        # 6. 状态迁移：PROCESSING → COMPLETED / FAILED
        if result:
            task_manager.complete_task(task_id, result.save_result_path)
        else:
            task_manager.fail_task(task_id, "Generation failed")

    except Exception as e:
        task_manager.fail_task(task_id, str(e))
    finally:
        task_manager.release_processing_lock(task_id)
```

### 6.5 任务取消机制

取消使用 `threading.Event` 作为信号：

```python
def cancel_task(self, task_id: str) -> bool:
    task = self._tasks[task_id]
    if task.status in [TaskStatus.COMPLETED, TaskStatus.FAILED]:
        return False           # 已结束的任务不可取消

    task.stop_event.set()       # 设置取消信号
    task.status = TaskStatus.CANCELLED
```

`stop_event` 在推理过程中被检查：
- `_process_single_task` 在推理前检查
- `generate_with_stop_event` 在处理图片/音频前检查
- 推理过程本身（`runner.run_pipeline`）不检查——这意味着**一旦推理开始，取消请求需要等到当前推理完成才能生效**

### 6.6 队列清理

```python
def _cleanup_old_tasks(self, keep_count: int = 1000):
    """保留最多 1000 个任务记录，超出时清理最早完成的任务"""
    if len(self._tasks) <= keep_count:
        return
    completed_tasks = [(id, t) for id, t in self._tasks.items()
                       if t.status in [COMPLETED, FAILED, CANCELLED]]
    completed_tasks.sort(key=lambda x: x[1].end_time or x[1].start_time)
    for task_id, _ in completed_tasks[:len(self._tasks) - keep_count]:
        del self._tasks[task_id]
```

## 7. DistributedInferenceService：推理服务

**核心文件**：`lightx2v/server/services/inference/service.py`

这是连接 API 层和底层 Runner 的桥梁。

```python
class DistributedInferenceService:
    def __init__(self):
        self.worker = None          # TorchrunInferenceWorker
        self.is_running = False

    def start_distributed_inference(self, args) -> bool:
        self.worker = TorchrunInferenceWorker()
        self.worker.init(args)      # 初始化分布式环境 + 加载模型
        self.is_running = True

    async def submit_task_async(self, task_data: dict) -> Optional[dict]:
        """提交推理任务（仅 Rank 0 调用）"""
        self.worker.processing = True
        result = await self.worker.process_request(task_data)
        self.worker.processing = False
        return result

    async def run_worker_loop(self):
        """非 Rank 0 进程的循环（等待 broadcast）"""
        if self.worker.rank != 0:
            await self.worker.worker_loop()
```

## 8. TorchrunInferenceWorker：推理执行者

**核心文件**：`lightx2v/server/services/inference/worker.py`

这是实际调用 `runner.run_pipeline()` 的地方。

### 8.1 初始化

```python
class TorchrunInferenceWorker:
    def init(self, args) -> bool:
        # 1. 初始化分布式环境
        if self.world_size > 1:
            self.dist_manager.init_process_group()

        # 2. 构建 config
        config = set_config(args)
        if config["parallel"]:
            set_parallel_config(config)

        # 3. 创建 Runner（加载模型）
        self.runner = init_runner(config)

        # 4. 创建可复用的 input_info 模板
        self.input_info = init_empty_input_info(args.task)
```

### 8.2 处理请求

```python
    async def process_request(self, task_data: dict) -> dict:
        # 1. [多卡] Rank 0 广播任务数据给其他 Rank
        if self.world_size > 1 and self.rank == 0:
            task_data = self.dist_manager.broadcast_task_data(task_data)

        # 2. 动态 LoRA 切换
        lora_name = task_data.pop("lora_name", None)
        lora_strength = task_data.pop("lora_strength", 1.0)
        if self.lora_dir:
            self.switch_lora(lora_name, lora_strength)

        # 3. 更新 input_info（复用模板，避免重复创建）
        update_input_info_from_dict(self.input_info, task_data)

        # 4. 更新 Runner config（如分辨率、步数等可能逐请求变化）
        self.runner.set_config(task_data)

        # 5. 执行推理
        self.runner.run_pipeline(self.input_info)

        # 6. [多卡] 同步
        if self.world_size > 1:
            self.dist_manager.barrier()

        return {"task_id": ..., "status": "success", "save_result_path": ...}
```

### 8.3 非 Rank 0 的 Worker 循环

```python
    async def worker_loop(self):
        """非 Rank 0 的无限循环：等待 → 接收 → 执行"""
        while True:
            # broadcast_task_data() 无参数 = 接收端
            task_data = self.dist_manager.broadcast_task_data()
            if task_data is None:
                break                  # 收到停止信号
            await self.process_request(task_data)
```

## 9. Generation Service：生成服务

**核心文件**：`lightx2v/server/services/generation/base.py`

Generation Service 负责**请求预处理**（图片下载、base64 解码、路径解析）和**结果封装**。实际推理委托给 `inference_service.submit_task_async()`。

```python
class BaseGenerationService(ABC):
    async def generate_with_stop_event(self, message, stop_event):
        # 1. 从 Pydantic Model 提取字段为 dict
        task_data = {field: getattr(message, field)
                     for field in message.model_fields_set}

        # 2. 处理图片（URL → 下载, base64 → 解码保存, 本地路径 → 直接使用）
        if message.image_path:
            await self._process_image_path(message.image_path, task_data)

        # 3. 处理掩码图片（如果有，需要与原图对齐分辨率）
        if message.image_mask_path:
            await self._process_image_mask_path(message.image_mask_path, task_data)
            self._pack_image_and_mask_as_dir(task_data)  # 打包成目录

        # 4. 处理音频（URL/base64/本地路径）
        if message.audio_path:
            await self._process_audio_path(message.audio_path, task_data)

        # 5. 准备输出路径
        self._prepare_output_path(message.save_result_path, task_data)

        # 6. 提交给推理服务
        result = await self.inference_service.submit_task_async(task_data)

        # 7. 封装响应
        if result.get("status") == "success":
            return TaskResponse(task_id=..., task_status="completed", ...)
        else:
            raise RuntimeError(result.get("error"))
```

**Video vs Image 的差异**：

| 维度 | VideoGenerationService | ImageGenerationService |
|------|----------------------|----------------------|
| 输出扩展名 | `.mp4` | `.png` |
| 任务类型 | `t2v,i2v,s2v` | `t2i,i2i` |
| 额外处理 | audio_path, talk_objects, target_fps | aspect_ratio |

## 11. DistributedManager：多卡通信

**核心文件**：`lightx2v/server/services/distributed_utils.py`

### 11.1 任务数据广播

多卡推理时，只有 Rank 0 接收 HTTP 请求。其他 Rank 通过 `broadcast_task_data` 获取任务数据。

```python
def broadcast_task_data(self, task_data=None):
    if self.is_rank_zero():
        # 发送端
        if task_data is None:
            stop_signal = torch.tensor([1])     # 停止信号
        else:
            stop_signal = torch.tensor([0])     # 正常任务
        dist.broadcast(stop_signal, src=0, group=self.task_pg)

        if task_data is not None:
            task_bytes = pickle.dumps(task_data)
            task_length = torch.tensor([len(task_bytes)])
            dist.broadcast(task_length, src=0, group=self.task_pg)
            self._broadcast_byte_chunks(task_bytes)  # 分块传输
        return task_data
    else:
        # 接收端
        stop_signal = torch.tensor([0])
        dist.broadcast(stop_signal, src=0, group=self.task_pg)
        if stop_signal.item() == 1:
            return None                         # 收到停止信号

        task_length = torch.tensor([0])
        dist.broadcast(task_length, src=0, group=self.task_pg)
        task_bytes = self._receive_byte_chunks(task_length.item())
        return pickle.loads(task_bytes)
```

**为什么用 gloo 而不用 nccl？**

任务数据是 Python dict（包含字符串、列表等），需要 pickle 序列化为字节流，然后用 `torch.uint8` tensor 传输。gloo 后端支持 CPU tensor 的 broadcast，而 nccl 只支持 CUDA tensor。任务数据量很小（几 KB），gloo 的性能足够。

**为什么分块传输？**（`_broadcast_byte_chunks`）

单次 broadcast 的 tensor 太大可能导致超时（gloo 的默认超时较短）。分块（1MB/块）可以避免这个问题。

## 13. 与 CLI 推理的对比

| 维度 | CLI 推理 (`lightx2v/infer.py`) | Server 推理 |
|------|-------------------------------|-------------|
| 入口 | `python -m lightx2v.infer --config_path ...` | `python -m lightx2v.server --model_path ...` |
| 生命周期 | 一次推理后退出 | 常驻进程，持续接受请求 |
| 模型加载 | 每次启动都加载 | 启动时加载一次，后续复用 |
| input_info | 从 config 构建 | 复用模板 + `update_input_info_from_dict` |
| 并发 | 无 | 队列 + 互斥锁，串行推理 |
| 图片输入 | 本地路径 | URL/base64/本地路径 |
| 输出 | 本地文件 | 本地文件 + HTTP 流式下载 |
| LoRA | 启动时固定 | 支持逐请求动态切换 |
| 取消 | Ctrl+C | API 取消（`stop_event`） |
| 监控 | 无 | Prometheus 指标 |

## 14. Metrics 监控

**核心文件**：`lightx2v/server/metrics/metrics.py`、`monitor.py`

框架使用 Prometheus 采集推理各阶段的耗时指标：

| 指标名 | 类型 | 说明 |
|--------|------|------|

| `lightx2v_worker_request_success` | Counter | 成功请求数 |
| `lightx2v_worker_request_failure` | Counter | 失败请求数（按 error_type 分标签） |

## 15. 思考题

1. **为什么 `_task_processing_loop` 用单独的守护线程而不是 FastAPI 的 BackgroundTasks？**
   - 提示：考虑 PyTorch GPU 推理与 asyncio 事件循环的关系

2. **如果两个请求几乎同时到达，第二个请求的状态变化是什么？**
   - 提示：跟踪 `create_task` → `get_next_pending_task` → `acquire_processing_lock` 的流程

3. **`stop_event` 能否真正中断正在运行的 `runner.run_pipeline()`？如果不能，需要怎么改？**
   - 提示：考虑在 scheduler 的 denoising loop 中检查 stop_event

4. **DistributedManager 用 pickle + gloo broadcast 传输任务数据。如果任务数据包含大型图片 tensor，会有什么问题？**
   - 提示：考虑传输的是 `task_data` dict 的内容——图片路径（字符串）还是图片数据（bytes）

5. **server_config 是模块级全局单例，在多进程（torchrun）环境下，每个进程有独立的 server_config 吗？**
   - 提示：考虑 Python 的 fork vs spawn 进程模型
