## 1. 框架解决的核心问题

LightX2V 要解决的核心问题是：**如何在统一框架下支持多种视频生成模型，同时提供灵活的优化手段？**

这带来几个设计挑战：
1. 不同模型架构差异大（Wan、HunyuanVideo、Qwen 等）
2. 优化手段多样（Offload、量化、缓存、并行）
3. 需要易于扩展新模型和新优化

LightX2V 的解决方案是：**分层架构 + 注册机制 + 配置驱动**

---

## 2. 三层架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                      Pipeline 层                             │
│  (LightX2VPipeline - 用户接口，配置管理，Runner 调度)         │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                       Runner 层                              │
│  (DefaultRunner/WanRunner/... - 组件编排，推理流程控制)       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                     Components 层                            │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐       │
│  │ Encoders │ │   DiT    │ │Scheduler │ │   VAE    │       │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 Pipeline 层：用户接口

**职责**：
- 提供简洁的用户 API
- 管理配置参数
- 根据 model_cls 选择对应的 Runner

**核心文件**：`lightx2v/pipeline.py`

```python
# pipeline.py:63-82 - Pipeline 初始化
@dict_like
class LightX2VPipeline:
    def __init__(
        self,
        task,           # 任务类型：t2v, i2v, t2i, i2i
        model_path,     # 模型路径
        model_cls,      # 模型类型：wan2.1, hunyuan_video_1.5, qwen_image 等
        ...
    ):
        self.task = task
        self.model_path = model_path
        self.model_cls = model_cls
        ...
```

**设计要点**：
- `@dict_like` 装饰器让 Pipeline 可以像字典一样访问属性，方便配置传递
- `model_cls` 是路由到具体 Runner 的关键

### 2.2 Runner 层：组件编排

**职责**：
- 加载和管理各个组件（编码器、DiT、VAE）
- 控制推理流程（encode → denoise → decode）
- 处理不同任务类型的差异

**继承体系**：
```
BaseRunner (抽象基类，定义接口)
    │
    └── DefaultRunner (通用实现，推理循环)
            │
            ├── WanRunner
            ├── HunyuanVideo15Runner
            ├── QwenImageRunner
            └── ...
```

**核心文件**：
- `lightx2v/models/runners/base_runner.py` - 接口定义
- `lightx2v/models/runners/default_runner.py` - 通用实现
- `lightx2v/models/runners/wan/wan_runner.py` - Wan 模型实现

**什么是生命周期钩子？**

生命周期钩子（Lifecycle Hooks）是一种设计模式，指的是在程序执行的特定阶段预留的"插入点"，让子类可以在这些时机插入自定义逻辑。

用大白话说就是：**框架在关键节点喊一声"到这儿了，有人要加戏吗？"**

```
# 推理的生命周期
run_pipeline()
    │
    ├── init_run()           # 钩子1：推理开始前，要初始化啥？
    │
    ├── for segment in segments:
    │       │
    │       ├── init_run_segment()   # 钩子2：每段开始前
    │       │
    │       ├── run_segment()        # 实际去噪
    │       │
    │       └── end_run_segment()    # 钩子3：每段结束后
    │
    └── end_run()            # 钩子4：推理结束，清理资源
```

**BaseRunner 定义了这些钩子（空实现）**：
```python
def init_run(self):
    pass  # 默认啥也不干

def end_run(self):
    pass  # 默认啥也不干
```

**子类可以覆盖它们**：
```python
class WanRunner(DefaultRunner):
    def init_run(self):
        # Wan 模型需要在推理前做一些特殊初始化
        self.prepare_wan_specific_stuff()
```

类似的概念在很多地方都有：React 的 `useEffect`、Vue 的 `mounted`、Git 的 `pre-commit` 等。

### 初始化流程

```python
class DefaultRunner(BaseRunner):
```

**什么是 Prompt Enhancer？**

`has_prompt_enhancer` 用于检测是否有"提示词增强"服务可用。

Prompt Enhancer 是一个独立的 LLM 服务（默认用 Qwen2.5-32B），作用是把用户的简短提示词扩写成详细的视频生成描述。

```

↓ Prompt Enhancer 扩写

across a muddy field, dribbling a soccer ball with precise footwork.
The player glances toward the goalpost, adjusts their stance, and
kicks the ball forcefully into the net. Raindrops fall lightly,
creating reflections under stadium floodlights. The camera follows
the ball's trajectory in a smooth pan.
```

**工作流程**：

```python
# default_runner.py:469-470
def run_pipeline(self, input_info):
```

**为什么是远程服务？**

因为 Prompt Enhancer 需要一个 32B 的 LLM，显存占用大。把它部署成独立服务，可以：
1. 与视频生成模型分开部署，不抢显存
2. 多个视频生成实例共享一个 Enhancer ��务
3. 可选启用，不需要时不占资源

### run_main()：主推理流程

```python
# default_runner.py:362-393
@ProfilingContext4DebugL2("Run DiT")
def run_main(self):
    """主推理流程：分段去噪 + VAE 解码"""

    # 1. 初始化
    self.init_run()

    # 2. 分段推理（支持长视频）
    for segment_idx in range(self.video_segment_num):
        # 2.1 分段初始化
        self.init_run_segment(segment_idx)

        # 2.2 去噪循环
        latents = self.run_segment(segment_idx)

        # 2.3 VAE 解码
        if self.config.get("use_stream_vae", False):
            # 流式解码（省显存）
            frames = []
            for frame_segment in self.run_vae_decoder_stream(latents):
                frames.append(frame_segment)
            self.gen_video = torch.cat(frames, dim=2)
        else:
            self.gen_video = self.run_vae_decoder(latents)

        # 2.4 分段结束
        self.end_run_segment(segment_idx)

    # 3. 后处理（插帧、保存等）
    gen_video_final = self.process_images_after_vae_decoder()

    # 4. 清理
    self.end_run()

    return gen_video_final
```

### run_segment()：去噪循环

```python
# default_runner.py:175-208
def run_segment(self, segment_idx=0):
    """单个分段的去噪循环"""
    infer_steps = self.model.scheduler.infer_steps

    for step_index in range(infer_steps):
        # 检查停止信号
        if self.video_segment_num == 1:
            self.check_stop()

        logger.info(f"==> step_index: {step_index + 1} / {infer_steps}")

        # 三阶段推理
        with ProfilingContext4DebugL1("step_pre"):
            self.model.scheduler.step_pre(step_index=step_index)

        with ProfilingContext4DebugL1("🚀 infer_main"):
            self.model.infer(self.inputs)  # DiT 前向

        with ProfilingContext4DebugL1("step_post"):
            self.model.scheduler.step_post()

        # 进度回调
        if self.progress_callback:
            progress = (step_index + 1) / infer_steps * 100
            self.progress_callback(progress, 100)

    return self.model.scheduler.latents
```

**核心循环**：`step_pre → model.infer → step_post`，这是扩散模型去噪的标准流程。

### step_pre 和 step_post 详解

这两个方法是调度器（Scheduler）的核心，分别在每个去噪步骤的前后执行。

**step_pre：准备当前步骤的输入**

```python
# scheduler.py:16-19 (BaseScheduler)
def step_pre(self, step_index):
    self.step_index = step_index  # 记录当前步骤索引
    if GET_DTYPE() == GET_SENSITIVE_DTYPE():
        self.latents = self.latents.to(GET_DTYPE())  # 转换数据类型

# wan/scheduler.py:328-332 (WanScheduler)
def step_pre(self, step_index):
    super().step_pre(step_index)
    # 准备当前步骤的时间步输入
    self.timestep_input = torch.stack([self.timesteps[self.step_index]])
    # Wan2.2 I2V 特殊处理：根据 mask 调整时间步
    if self.config["model_cls"] == "wan2.2" and self.config["task"] in ["i2v"]:
        self.timestep_input = (self.mask[0][:, ::2, ::2] * self.timestep_input).flatten()
```

**step_pre 做的事情**：
1. 记录当前步骤索引
2. 准备时间步 `timestep_input`（告诉模型当前是第几步）
3. 可能的数据类型转换

---

**step_post：根据模型输出更新 latents**

这是去噪的核心数学计算，不同调度器实现不同。

**简单版本（HunyuanVideo - Euler 方法）**：
```python
# hunyuan_video/scheduler.py:136-140
def step_post(self):
    model_output = self.noise_pred.to(torch.float32)  # 模型预测的噪声
    sample = self.latents.to(torch.float32)           # 当前 latent
    dt = self.sigmas[self.step_index + 1] - self.sigmas[self.step_index]  # 时间步长
    self.latents = sample + model_output * dt         # 一阶欧拉更新
```

这就是最简单的 ODE 求解：`x_{t+1} = x_t + f(x_t) * dt`

**复杂版本（Wan - UniPC 多步方法）**：
```python
# wan/scheduler.py:334-374
def step_post(self):
    model_output = self.noise_pred.to(torch.float32)
    sample = self.latents.to(torch.float32)

    # 1. 转换模型输出格式
    model_output_convert = self.convert_model_output(model_output, sample=sample)

    # 2. 校正步骤（Corrector）：用历史信息修正当前预测
    if use_corrector:
        sample = self.multistep_uni_c_bh_update(...)

    # 3. 更新历史记录（用于多步方法）
    self.model_outputs[-1] = model_output_convert
    self.timestep_list[-1] = timestep

    # 4. 预测步骤（Predictor）：计算下一个 latent
    prev_sample = self.multistep_uni_p_bh_update(...)

    # 5. 更新 latents
    self.latents = prev_sample
```

**step_post 做的事情**：
1. 获取模型预测的噪声 `noise_pred`
2. 根据调度算法（Euler/DDIM/UniPC/...）计算下一步的 latent
3. 更新 `self.latents`

---

**不同调度器的 step_post 对比**：

| 调度器 | 算法 | 特点 |
|--------|------|------|
| HunyuanVideo15Scheduler | Euler | 简单，一阶精度 |
| WanScheduler | UniPC | 多步方法，二阶精度，需要历史信息 |
| 蒸馏模型调度器 | 简化版 | 步数少，跳过某些计算 |

---

**形象理解**：

把去噪过程想象成从山顶（纯噪声）走到山脚（清晰图像）：

```
step_pre:  "我现在在第 N 个台阶，准备好下一步要用的信息"
model.infer: "模型告诉我往哪个方向走（预测噪声/速度）"
step_post: "根据方向，实际走一步，更新我的位置（latents）"
```

```python
@ProfilingContext4DebugL2("Run Encoders")
def _run_input_encoder_local_i2v(self):
```

### end_run()：资源清理

```python
# default_runner.py:219-243
def end_run(self):
    """推理结束后的资源清理"""
    # 1. 清理调度器状态
    self.model.scheduler.clear()

    # 2. 清理输入
    if hasattr(self, "inputs"):
        del self.inputs
    self.input_info = None

    # 3. lazy_load 模式下卸载模型
    if self.config.get("lazy_load", False) or self.config.get("unload_modules", False):
        if hasattr(self.model.transformer_infer, "offload_manager"):
            del self.model.transformer_infer.offload_manager
        del self.model

    # 4. 清理显存
    torch_device_module.empty_cache()
    gc.collect()
```

## 3. 注册机制：解耦的关键

### 3.1 Register 类是什么？

Register 本质上是一个**可以当装饰器用的字典**。它的作用是：把"名字"和"类"关联起来，方便后续通过名字找到对应的类。

```python
# 目标效果
RUNNER_REGISTER = Register()

# 注册：把 "wan2.1" 和 WanRunner 类关联
@RUNNER_REGISTER("wan2.1")
class WanRunner:
    pass

# 使用：通过名字拿到类
runner_cls = RUNNER_REGISTER["wan2.1"]  # 得到 WanRunner 类
runner = runner_cls(config)              # 实例化
```

### 3.2 Register 类实现详解

**核心文件**：`lightx2v/utils/registry_factory.py`

```python
class Register(dict):
    def __init__(self, *args, **kwargs):
        super(Register, self).__init__(*args, **kwargs)
        self._dict = {}  # 真正存数据的地方
```

继承 `dict` 但实际用 `self._dict` 存数据。

---

**核心方法：`__call__`（让实例可以像函数一样调用）**

```python
def __call__(self, target_or_name):
    if callable(target_or_name):
        # 情况1：直接装饰，没有参数
        # @RUNNER_REGISTER
        # class WanRunner: ...
        return self.register(target_or_name)
    else:
        # 情况2：带参数装饰
        # @RUNNER_REGISTER("wan2.1")
        # class WanRunner: ...
        return lambda x: self.register(x, key=target_or_name)
```

这里处理了两种装饰器用法：

**用法1：不带参数**
```python
@RUNNER_REGISTER
class WanRunner:
    pass

# 等价于：
class WanRunner:
    pass
WanRunner = RUNNER_REGISTER(WanRunner)
# target_or_name 是 WanRunner 类，callable 为 True
# 用类名 "WanRunner" 作为 key
```

**用法2：带参数（常用）**
```python
@RUNNER_REGISTER("wan2.1")
class WanRunner:
    pass

# 等价于：
class WanRunner:
    pass
decorator = RUNNER_REGISTER("wan2.1")  # 返回一个 lambda
WanRunner = decorator(WanRunner)        # lambda 调用 register
# target_or_name 是字符串 "wan2.1"，callable 为 False
# 用 "wan2.1" 作为 key
```

---

**register 方法：实际的注册逻辑**

```python
def register(self, target, key=None):
    # target 必须是可调用的（类或函数）
    if not callable(target):
        raise Exception(f"Error: {target} must be callable!")

    # 如果没指定 key，用类名
    if key is None:
        key = target.__name__

    # 不允许重复注册
    if key in self._dict:
        raise Exception(f"{key} already exists.")

    # 存入字典
    self[key] = target

    # 返回原类（装饰器必须返回原对象）
    return target
```

---

**字典操作方法**

```python
def __setitem__(self, key, value):
    self._dict[key] = value

def __getitem__(self, key):
    return self._dict[key]

def __contains__(self, key):
    return key in self._dict
```

这些让 Register 可以像普通字典一样使用：
```python
RUNNER_REGISTER["wan2.1"] = WanRunner  # __setitem__
cls = RUNNER_REGISTER["wan2.1"]        # __getitem__
"wan2.1" in RUNNER_REGISTER            # __contains__
```

---

### 3.3 完整流程图

```
1. 定义阶段（import 时执行）
   @RUNNER_REGISTER("wan2.1")
   class WanRunner: ...
        │
        ▼
   RUNNER_REGISTER("wan2.1") 被调用
        │
        ▼

2. 使用阶段（运行时）

        │

        │
        ▼
```


### 3.5 注册的触发：pipeline.py 中的 import

注册机制有一个关键细节：**装饰器是在 import 时执行的**。

看 `pipeline.py` 顶部的 import：

```python
# pipeline.py:13-28
from lightx2v.models.runners.hunyuan_video.hunyuan_video_15_runner import HunyuanVideo15Runner  # noqa: F401
from lightx2v.models.runners.wan.wan_runner import Wan22MoeRunner, WanRunner  # noqa: F401
from lightx2v.models.runners.qwen_image.qwen_image_runner import QwenImageRunner  # noqa: F401
# ... 更多 Runner
```

**这些 import 看起来没有被使用**（注意 `# noqa: F401` 告诉 linter 别报警告），但它们的真正目的是**触发装饰器执行**。

**执行流程**：

```
用户: from lightx2v import LightX2VPipeline
        │
        ▼
Python 加载 pipeline.py
        │
        ▼
执行 pipeline.py 顶部的 import 语句
        │
        ├── import WanRunner
        │       │
        │       ▼
        │   Python 加载 wan_runner.py，执行类定义
        │       │
        │       ▼
        │   @RUNNER_REGISTER("wan2.1") 装饰器执行
        │       │
        │       ▼
        │   RUNNER_REGISTER["wan2.1"] = WanRunner  ← 注册完成！
        │
        ├── import HunyuanVideo15Runner → 同理注册
        │
        └── ... 所有 Runner 都被注册
        │
        ▼
后续使用：RUNNER_REGISTER["wan2.1"] → 返回 WanRunner 类
```

**添加新模型时的完整步骤**：

```python
# 步骤1：新建 runner 文件，用装饰器注册
# lightx2v/models/runners/my_model/my_runner.py
@RUNNER_REGISTER("my_model")
class MyRunner(DefaultRunner):
    def load_transformer(self):
        ...

# 步骤2：在 pipeline.py 中添加 import（触发注册）
from lightx2v.models.runners.my_model.my_runner import MyRunner  # noqa: F401
```

如果忘了步骤2，Runner 不会被注册，运行时会报 `KeyError: 'my_model'`。

### 3.6 RUNNER_REGISTER 是全局唯一的吗？

**Register 类不是单例，但 RUNNER_REGISTER 是全局唯一实例。**

`Register` 类本身可以创建多个实例：

```python
# registry_factory.py:59-71
MM_WEIGHT_REGISTER = Register()           # 实例1
ATTN_WEIGHT_REGISTER = Register()         # 实例2
RUNNER_REGISTER = Register()              # 实例3
# ... 每个注册器是独立的实例
```

但每个具体的注册器（如 `RUNNER_REGISTER`）是**模块级变量**，由于 Python 的模块导入机制，模块只会被加载一次：

```python
# 无论从哪里 import，拿到的都是同一个对象
from lightx2v.utils.registry_factory import RUNNER_REGISTER  # 文件A
from lightx2v.utils.registry_factory import RUNNER_REGISTER  # 文件B
# 两个 RUNNER_REGISTER 是同一个对象（id 相同）
```

这种模式叫**模块单例**（Module Singleton），利用 Python 模块只加载一次的特性来保证全局唯一。

| 概念 | 是否单例 |
|------|----------|
| `Register` 类 | ❌ 不是，可以创建多个实例 |
| `RUNNER_REGISTER` 变量 | ✅ 全局唯一（模块级变量） |

### 3.7 为什么这样设计？

**好处**：
1. **解耦**：Pipeline 不需要知道 Runner 的具体实现，只需要知道名字
2. **可扩展**：新增模型只需要加装饰器 + import，不用改 Pipeline 逻辑
3. **配置驱动**：可以通过配置文件的字符串选择实现

**实际使用**：
```python
# pipeline.py:447
def _init_runner(self, config):
    # config["model_cls"] = "wan2.1"
    runner = RUNNER_REGISTER[config["model_cls"]](config)
    #        ↑ 拿到 WanRunner 类              ↑ 实例化
    return runner
```

### 4.3 LockableDict：防止意外修改

**核心文件**：`lightx2v/utils/lockable_dict.py`

LockableDict 是一个**可以锁定的字典**。锁定后，任何修改操作都会抛出异常。

**为什么需要它？**

配置在初始化阶段会被多处代码读取和设置，但一旦初始化完成，配置就不应该再被修改了。如果运行时意外修改了配置，可能导致难以排查的 bug。

```python
# default_runner.py:96
self.config.lock()  # 初始化完成后锁定配置
```

**基本用法**：

```python
from lightx2v.utils.lockable_dict import LockableDict

# 创建
config = LockableDict({"a": 1, "b": 2})

# 正常使用（未锁定）
config["a"] = 10  # OK
config.a = 10     # OK，支持属性访问

# 锁定
config.lock()

# 锁定后修改会报错
config["a"] = 20  # TypeError: Dictionary is locked, current operation not allowed.
```

**临时解锁**：

有时候确实需要在锁定后修改配置，可以用 `temporarily_unlocked()`：

```python
# default_runner.py:169
def set_config(self, config_modify):
    with self.config.temporarily_unlocked():
        self.config.update(config_modify)
    # 退出 with 块后自动恢复锁定状态
```

**核心实现**：

```python
class LockableDict(dict):
    def __init__(self, *args, **kwargs):
        self._locked = False  # 锁定状态

    def lock(self):
        self._locked = True

    def unlock(self):
        self._locked = False

    def __setitem__(self, key, value):
        if self._locked:
            raise TypeError("Dictionary is locked")
        dict.__setitem__(self, key, value)

    # __delitem__, clear, pop, update 等方法同理
```

**特性**：
1. **递归锁定**：嵌套的字典也会被自动转换为 LockableDict 并一起锁定
2. **属性访问**：支持 `config.key` 的写法，等价于 `config["key"]`
3. **临时解锁**：`with config.temporarily_unlocked()` 退出后自动恢复锁定

## 8. 思考题

1. **为什么使用注册机制而不是简单的 if-else？**
   - 提示：考虑扩展性和代码组织

   开闭原则：新增模型不需要修改 Pipeline 代码，只需要加一个 @RUNNER_REGISTER("xxx") 装饰器
   
   代码组织：避免 Pipeline 里出现一大堆 if model_cls == "wan2.1": ... elif model_cls == "hunyuan": ...

   解耦：Pipeline 不需要 import 所有 Runner，Runner 自己注册自己

2. **Pipeline 层和 Runner 层的职责边界在哪里？**
   - 提示：Pipeline 管理什么？Runner 管理什么？

   pipeline负责路由到模型对应的实现上，同时整合完整配置，runner负责管理实际模型推理流程

3. **如果要添加一个新的视频生成模型，需要修改哪些文件？**
   - 提示：参考 WanRunner 的实现

    适配一个全新模型，需要在三层架构上分别新增对应实现：

    **必须新建的（三层各一个）**：
    - Runner 层：`lightx2v/models/runners/xxx/xxx_runner.py`，继承 DefaultRunner，实现 `load_transformer`、`load_text_encoder`、`load_vae`、`init_scheduler` 等方法
    - Network 层：`lightx2v/models/networks/xxx/`，实现该模型的 DiT 网络结构（不同模型的 Transformer 架构不同，无法复用）
    - Scheduler：`lightx2v/models/schedulers/xxx/`，实现该模型的去噪调度算法（不同模型用不同的采样策略）

    **必须修改的**：
    - `pipeline.py`：添加 import 语句触发 `@RUNNER_REGISTER` 装饰器执行注册
    - `pipeline.py __init__`：如果新模型的 `vae_stride`、`num_channels_latents` 等参数与现有模型不同，需要添加对应的 `elif` 分支
    - `pipeline.py enable_offload/enable_quantize`：如果新模型使用了不同的编码器（如既不是 T5 也不是 Qwen），需要添加对应的配置映射

    **可能需要的**：
    - 新建编码器：如果模型使用了框架中尚未支持的文本/图像编码器
    - 新建配置文件：`configs/xxx/`
    



4. **配置为什么要在初始化后锁定？**
   - 提示：考虑多线程和调试场景

    多线程安全：服务化部署时多个请求可能共享 Runner，锁定配置避免竞态条件

    调试友好：如果配置被意外修改，锁定机制会立即抛异常，而不是产生奇怪的行为

    防止意外修改：初始化后配置应该是"只读"的，如果运行时被意外修改，可能导致难以排查的 bug
