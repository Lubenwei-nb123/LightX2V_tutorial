## 1. 输入编码器概览

### 1.1 不同模型的编码器选型

| 模型 | 文本编码器 | 图像编码器 | VAE |
|------|-----------|-----------|-----|
| Wan 2.1 | T5 (umt5-xxl) | CLIP (xlm-roberta) | WanVAE |

| Z-Image | Qwen3 | 无 | Z-Image VAE |

**为什么不同模型用不同的编码器？**

每个视频生成模型在训练时就绑定了特定的编码器——DiT 的 cross-attention 权重是对着编码器的输出特征空间训练的。换一个编码器，特征空间不匹配，生成质量会崩溃。所以推理框架必须为每个模型提供对应的编码器实现。

### 1.3 编码器的统一使用模式

虽然编码器的内部实现各不相同（T5 是 Transformer、CLIP 是 ViT、VAE 是 3D CNN），但从 Runner 的视角看，它们都遵循相同的使用模式：

```
1. 加载阶段：Runner.load_text_encoder() / load_image_encoder() / load_vae()
   │  读取 Offload 配置 → 决定设备
   │  读取量化配置 → 决定权重格式
   │  创建编码器实例
   │
2. 使用阶段：Runner.run_text_encoder() / run_image_encoder() / run_vae_encoder()
   │  [lazy_load] 加载
   │  调用 encoder.infer() 或 encoder.encode()
   │  [lazy_load] 卸载
   │
3. 输出格式：
   │  text_encoder → {"context": tensor, "context_null": tensor}
   │  image_encoder → tensor（CLIP 特征）或 (tensor, mask)（SigLIP）
   │  vae_encoder → latent tensor
```

## 2. 编码器的 Offload 与量化

### 2.1 Offload 的实现模式

所有编码器的 Offload 都遵循相同的模式（以 T5 为例）：

```python
# wan_runner.py:119-157
def load_text_encoder(self):
    # 1. 读取 Offload 配置
    t5_offload = self.config.get("t5_cpu_offload", self.config.get("cpu_offload"))
    if t5_offload:
        t5_device = torch.device("cpu")     # Offload：先加载到 CPU
    else:
        t5_device = torch.device(AI_DEVICE) # 不 Offload：直接加载到 GPU

    # 2. 创建编码器，传入 cpu_offload 标志
    text_encoder = T5EncoderModel(
        device=t5_device,
        cpu_offload=t5_offload,  # 编码器内部会根据这个标志管理设备迁移
        ...
    )
```

编码器内部的 Offload 行为：
- `cpu_offload=False`：权重常驻 GPU，`infer()` 直接计算
- `cpu_offload=True`：权重存在 CPU，`infer()` 时先 `.to(GPU)`，计算完再 `.to(CPU)`

T5 编码器甚至支持 **block 级 Offload**（和 DiT 类似的双缓冲机制）：

```python
# input_encoders/hf/wan/t5/model.py:52-68
class T5OffloadBlocksWeights(WeightModule):
    def __init__(self, block_nums, mm_type, lazy_load=False, ...):
        # 和 DiT 的 WanTransformerWeights 一样的 Offload buffer 机制
        self.offload_block_buffers = WeightModuleList(
            [T5OffloadSelfAttention(i, mm_type, create_cuda_buffer=True) for i in range(1)]
        )
        self.blocks = WeightModuleList(
            [T5OffloadSelfAttention(i, mm_type) for i in range(block_nums)]
        )
```

### 2.2 量化的实现模式

编码器也支持多种量化方案，复用了 DiT 层的注册器机制：

```python
# wan_runner.py:128-140
t5_quantized = self.config.get("t5_quantized", False)
if t5_quantized:
    t5_quant_scheme = self.config.get("t5_quant_scheme", None)
    # 根据量化方案选择对应的权重文件名
    t5_model_name = f"models_t5_umt5-xxl-enc-{tmp_t5_quant_scheme}.pth"
    t5_quantized_ckpt = find_torch_model_path(self.config, "t5_quantized_ckpt", t5_model_name)
else:
    t5_model_name = "models_t5_umt5-xxl-enc-bf16.pth"
    t5_original_ckpt = find_torch_model_path(self.config, "t5_original_ckpt", t5_model_name)
```

T5 内部的 Attention Block 也使用了 `MM_WEIGHT_REGISTER` 和 `RMS_WEIGHT_REGISTER`：

```python
# input_encoders/hf/wan/t5/model.py:71-80
class T5OffloadSelfAttention(WeightModule):
    def __init__(self, block_index, mm_type, ...):
        # 和 DiT 的 WanSelfAttention 完全一样的模式
        self.add_module("norm1",
            RMS_WEIGHT_REGISTER["sgl-kernel"](f"blocks.{block_index}.norm1.weight", ...))
        self.add_module("self_attn_q",
            MM_WEIGHT_REGISTER[mm_type](f"blocks.{block_index}.self_attn.q.weight", ...))
```

**编码器与 DiT 共享同一套权重注册器体系**——这意味着给 DiT 添加的 FP8/INT8 量化实现，编码器也能直接复用。

### 2.3 lazy_load 模式下的编码器

第3课讲过，lazy_load 模式下编码器在 `run_text_encoder()` 中即时加载和卸载：

## 3. 调度器接口设计

### 3.1 BaseScheduler：调度器基类

```python
# schedulers/scheduler.py:4-22
class BaseScheduler:
    def __init__(self, config):
        self.config = config
        self.latents = None          # 当前 latent（每个 step 更新）
        self.step_index = 0          # 当前步骤索引
        self.infer_steps = config["infer_steps"]  # 总步数
        self.infer_condition = True  # CFG 状态（True=条件推理，False=无条件）
        self.noise_pred = None       # model.infer() 的输出写到这里

    def step_pre(self, step_index):
        """每个 step 开始前调用"""
        self.step_index = step_index
        if GET_DTYPE() == GET_SENSITIVE_DTYPE():
            self.latents = self.latents.to(GET_DTYPE())

    def clear(self):
        """推理结束后清理状态"""
        pass
```

**调度器是 Runner 和 Model 之间的共享状态容器**：

```
Runner 写入 → Scheduler ← Model 读/写
    │                           │
    │  prepare()                │  读 self.latents（当前 latent）
    │  step_pre()               │  读 self.timestep_input
    │  step_post()              │  写 self.noise_pred（预测噪声）
    │                           │  读 self.infer_condition（CFG 状态）
```

### 3.2 调度器的三个核心方法

| 方法 | 调用者 | 职责 |
|------|--------|------|
| `prepare()` | `DefaultRunner.init_run()` | 生成初始噪声、计算 timesteps 序列 |
| `step_pre()` | `DefaultRunner.run_segment()` | 准备当前步的输入（timestep） |
| `step_post()` | `DefaultRunner.run_segment()` | 用 noise_pred 更新 latents |

对应的调用时序：

```python
# default_runner.py:343-361 → init_run
self.model.scheduler.prepare(
    seed=self.input_info.seed,
    latent_shape=self.input_info.latent_shape,
    image_encoder_output=self.inputs["image_encoder_output"]
)

# default_runner.py:175-208 → run_segment
for step_index in range(infer_steps):
    self.model.scheduler.step_pre(step_index=step_index)  # ①
    self.model.infer(self.inputs)                          # ②
    self.model.scheduler.step_post()                       # ③
```

## 4. 两种采样策略的对比

### 4.1 WanScheduler：UniPC（多步方法）

**理论背景**：UniPC（Unified Predictor-Corrector）是一种高阶 ODE 求解器。它利用历史步骤的信息来提高精度，类似于数值分析中的 Adams-Bashforth 方法。

**prepare()：初始化噪声和 timesteps**

```python
# wan/scheduler.py:31-54
def prepare(self, seed, latent_shape, image_encoder_output=None):
    # 1. 生成初始噪声
    self.prepare_latents(seed, latent_shape, dtype=torch.float32)

    # 2. 计算 sigma 序列（控制噪声强度的曲线）
    alphas = np.linspace(1, 1/self.num_train_timesteps, self.num_train_timesteps)[::-1]
    sigmas = 1.0 - alphas
    # 应用 shift（调整噪声强度分布）
    sigmas = self.shift * sigmas / (1 + (self.shift - 1) * sigmas)
    self.timesteps = sigmas * self.num_train_timesteps

    # 3. 初始化多步方法的历史缓存
    self.model_outputs = [None] * self.solver_order  # solver_order=2
    self.timestep_list = [None] * self.solver_order
    self.last_sample = None

    # 4. 设置推理步数的 timesteps
    self.set_timesteps(self.infer_steps, device=AI_DEVICE, shift=self.sample_shift)
```

**prepare_latents()：生成初始噪声**

```python
# wan/scheduler.py:56-69
def prepare_latents(self, seed, latent_shape, dtype=torch.float32):
    self.generator = torch.Generator(device=AI_DEVICE).manual_seed(seed)
    self.latents = torch.randn(
        latent_shape[0], latent_shape[1], latent_shape[2], latent_shape[3],
        dtype=dtype, device=AI_DEVICE, generator=self.generator,
    )
    # Wan2.2 I2V 特殊处理：首帧用 VAE 编码结果替换噪声
    if self.config["model_cls"] == "wan2.2" and self.config["task"] in ["i2v", ...]:
        self.mask = masks_like(self.latents, zero=True)
        self.latents = (1.0 - self.mask) * self.vae_encoder_out + self.mask * self.latents
        #              ↑ 首帧保留 VAE 编码    ↑ 其余帧用随机噪声
```

**step_pre()：准备当前步输入**

```python
# wan/scheduler.py:328-332
def step_pre(self, step_index):
    super().step_pre(step_index)
    # 把当前步的 timestep 包装成 tensor，供 DiT 的 time_embedding 使用
    self.timestep_input = torch.stack([self.timesteps[self.step_index]])
```

**step_post()：UniPC 去噪更新**

```python
# wan/scheduler.py:334-374
def step_post(self):
    model_output = self.noise_pred.to(torch.float32)  # DiT 预测的噪声
    sample = self.latents.to(torch.float32)

    # 1. 转换模型输出格式：noise → x0_pred
    model_output_convert = self.convert_model_output(model_output, sample=sample)
    #   x0_pred = sample - sigma_t * model_output
    #   直觉：从当前带噪样本减去预测噪声，得到对干净样本的估计

    # 2. Corrector（校正步，用当前预测修正上一步的结果）
    use_corrector = self.step_index > 0 and self.last_sample is not None
    if use_corrector:
        sample = self.multistep_uni_c_bh_update(
            this_model_output=model_output_convert,
            last_sample=self.last_sample,    # 上一步的 latent
            this_sample=sample,              # 当前 latent
            order=self.this_order,           # 校正阶数
        )

    # 3. 更新历史记录（供下一步的多步方法使用）
    self.model_outputs[-1] = model_output_convert
    self.timestep_list[-1] = timestep

    # 4. Predictor（预测步，计算下一个 latent）
    self.last_sample = sample
    prev_sample = self.multistep_uni_p_bh_update(
        model_output=model_output,
        sample=sample,
        order=self.this_order,
    )

    # 5. 更新 latents
    self.latents = prev_sample
```

**UniPC 的直觉理解**：

```
一阶（Euler）：       往预测方向走一步（简单但不准）
二阶（UniPC order=2）：不仅看当前预测，还看上一步的预测，
                      "两点定一线"，用插值得到更准确的方向

Predictor：用历史信息预测下一步
Corrector：用当前步的新信息修正预测结果
两者交替 → 更高精度
```

### 4.2 HunyuanVideo15Scheduler：Euler（一阶方法）

HunyuanVideo 使用最简单的 Euler 方法——Flow Matching 范式。

**prepare()：初始化**

```python
# hunyuan_video/scheduler.py:46-51
def prepare(self, seed, latent_shape, image_encoder_output=None):
    self.prepare_latents(seed, latent_shape, dtype=torch.bfloat16)
    self.set_timesteps(self.infer_steps, device=AI_DEVICE, shift=self.sample_shift)
    # HunyuanVideo 特有：准备条件 latent 和 mask
    self.multitask_mask = self.get_task_mask(self.config["task"], latent_shape[-3])
    self.cond_latents_concat, self.mask_concat = self._prepare_cond_latents_and_mask(...)
```

**set_timesteps()：线性 sigma + shift**

```python
# hunyuan_video/scheduler.py:65-79
def set_timesteps(self, num_inference_steps, device, shift):
    sigmas = torch.linspace(1, 0, num_inference_steps + 1)  # 线性分布

    # 应用 shift（SD3 风格的时间偏移）
    if shift != 1.0:
        sigmas = self.sd3_time_shift(sigmas, shift)

    self.sigmas = sigmas
    self.timesteps = (sigmas[:-1] * self.num_train_timesteps).to(dtype=torch.float32, device=device)

def sd3_time_shift(self, t, shift):
    return (shift * t) / (1 + (shift - 1) * t)
```

**step_post()：Euler 更新（极其简洁）**

```python
# hunyuan_video/scheduler.py:136-140
def step_post(self):
    model_output = self.noise_pred.to(torch.float32)
    sample = self.latents.to(torch.float32)
    dt = self.sigmas[self.step_index + 1] - self.sigmas[self.step_index]  # 时间步长
    self.latents = sample + model_output * dt  # 一阶欧拉：x_{t+1} = x_t + v * dt
```

**这就是 Flow Matching 的核心公式**：模型预测的不是"噪声"而是"速度场"（velocity），更新规则就是 `x += v * dt`，和物理中"位移 = 速度 × 时间"完全一样。

### 4.3 两种调度器的对比

| 维度 | WanScheduler (UniPC) | HunyuanVideo15Scheduler (Euler) |
|------|---------------------|---------------------------------|
| **理论基础** | 扩散模型 + UniPC 求解器 | Flow Matching |
| **模型预测目标** | 噪声 ε（需要 `convert_model_output` 转换） | 速度 v（直接使用） |
| **更新公式** | 二阶多步法（用历史信息） | 一阶欧拉 `x += v * dt` |
| **step_post 复杂度** | ~200 行（Predictor + Corrector） | 4 行 |
| **精度** | 高（二阶精度） | 低（一阶精度） |
| **latent 初始精度** | float32 | bfloat16 |
| **典型步数** | 30-50 步 | 30-50 步 |
| **timestep 分布** | 非均匀（shift 调整） | 线性 + shift |

**为什么 HunyuanVideo 可以用简单的 Euler？**

Flow Matching 模型在训练时就学到了更"直"的速度场——从噪声到数据的路径更接近直线。所以即使用简单的一阶 Euler 方法，轨迹也不会偏太多。而传统扩散模型的去噪路径比较弯曲，需要更高阶的求解器（如 UniPC）来准确跟踪。

### 4.4 shift 参数的作用

两个调度器都使用了 `shift` 参数来调整 sigma 分布：

```python
# 相同的 shift 公式（SD3 风格）
sigmas = shift * sigmas / (1 + (shift - 1) * sigmas)
```

直觉理解：

```
shift = 1.0：均匀分布，各步噪声强度线性递减
shift > 1.0：前期步骤的噪声变化更大，后期更细腻
             → 让模型在高噪声阶段"大刀阔斧"确定结构
             → 在低噪声阶段"精雕细琢"完善细节
shift < 1.0：反过来（很少使用）
```

Wan 模型的 `sample_shift` 通常设为 5.0-8.0，HunyuanVideo 通常设为 7.0-17.0。

## 5. VAE 编解码

### 5.1 VAE 在推理流程中的角色

```
输入图像 ──VAE Encoder──→ latent ──DiT 去噪──→ latent ──VAE Decoder──→ 输出视频
  (像素空间)               (latent 空间)                               (像素空间)
  480×832                  60×104                                      480×832
```

VAE 不参与去噪过程，它只负责像素空间和 latent 空间之间的转换。

### 5.2 WanVAE 的结构

```python
# video_encoders/hf/wan/vae.py
class WanVAE:
    def __init__(self, vae_path, device, parallel=False, cpu_offload=False, ...):
        # 加载预训练的 VAE 权重
        self.model = self._build_model(...)
        weights = load_weights(vae_path, cpu_offload=cpu_offload, ...)
        self.model.load_state_dict(weights)

    def encode(self, videos):
        """编码：像素 → latent"""
        # videos: [B, C, T, H, W]，值域 [-1, 1]
        # 返回: latent [B, latent_C, T', H', W']
        return self.model.encode(videos)

    def decode(self, latents):
        """解码：latent → 像素"""
        # latents: [B, latent_C, T', H', W']
        # 返回: videos [B, C, T, H, W]，值域 [-1, 1]
        return self.model.decode(latents)
```

VAE 内部使用 **3D Causal Convolution**——因果卷积确保时间维度上只依赖过去帧，不"偷看"未来帧：

```python
# video_encoders/hf/wan/vae.py:24-49
class CausalConv3d(nn.Conv3d):
    """因果 3D 卷积"""
    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        # 时间维度的 padding 只在前面加（因果性）
        self._padding = (
            self.padding[2], self.padding[2],  # W
            self.padding[1], self.padding[1],  # H
            2 * self.padding[0], 0,            # T：只在前面 pad，不在后面 pad
        )
        self.padding = (0, 0, 0)

    def forward(self, x, cache_x=None):
        padding = list(self._padding)
        if cache_x is not None:  # 流式解码时使用缓存
            x = torch.cat([cache_x, x], dim=2)
            padding[4] -= cache_x.shape[2]
        x = F.pad(x, padding)
        return super().forward(x)
```

### 5.3 VAE 的 Offload 与复用

VAE 的 Offload 和编码器类似——通过 `vae_cpu_offload` 配置控制：

```python
# wan_runner.py:166-187
def load_vae_encoder(self):
    vae_offload = self.config.get("vae_cpu_offload", self.config.get("cpu_offload"))
    if vae_offload:
        vae_device = torch.device("cpu")
    else:
        vae_device = torch.device(AI_DEVICE)
    return self.vae_cls(vae_path=..., device=vae_device, cpu_offload=vae_offload, ...)
```

第3课讲过，VAE encoder 和 decoder 在可能时会**复用同一实例**（因为编解码功能在同一个类中）。

### 5.4 流式 VAE 解码

DefaultRunner 支持流式解码，用于减少显存峰值：

```python
# default_runner.py:381-386
if self.config.get("use_stream_vae", False):
    frames = []
    for frame_segment in self.run_vae_decoder_stream(latents):
        frames.append(frame_segment)
    self.gen_video = torch.cat(frames, dim=2)
else:
    self.gen_video = self.run_vae_decoder(latents)
```

流式解码将 latent 按时间维度分块，逐块解码，避免一次性解码整个视频占用大量显存。VAE 内部的 `CausalConv3d` 通过 `cache_x` 参数支持分块解码——每块解码时保存边界的激活值，传给下一块使用，确保分块解码和整体解码结果一致。

## 6. 编码器与调度器的协作

### 6.1 完整数据流

以 Wan I2V 任务为例，串联编码器、调度器和 DiT：

```
用户输入：prompt + image + seed
    │
    ├── T5.infer(prompt) → context [512, 4096]          文本特征
    ├── CLIP.visual(image) → clip_fea [257, 4096]       图像语义特征
    ├── VAE.encode(image) → vae_out [16+4, 21, 60, 104] 图像 latent + mask
    │
    ▼
scheduler.prepare(seed, latent_shape, image_encoder_output)
    ├── randn(seed) → latents [16, 21, 60, 104]         初始噪声
    ├── set_timesteps(infer_steps, shift) → timesteps    噪声衰减计划
    │
    ▼
for step in range(infer_steps):  ← 去噪循环
    │
    ├── scheduler.step_pre(step)
    │     └── timestep_input = timesteps[step]
    │
    ├── model.infer(inputs)
    │     ├── pre_infer: latent+vae_out → patchify → x
    │     │              timestep → embed0
    │     │              context+clip_fea → cross-attn 条件
    │     ├── transformer_infer: 48 个 Block
    │     └── post_infer: unpatchify → noise_pred
    │     └── CFG: scheduler.noise_pred = 合并结果
    │
    └── scheduler.step_post()
          └── latents = 用 noise_pred 更新 latents

VAE.decode(latents) → video [3, 81, 480, 832]           最终视频
```

### 6.2 scheduler 作为共享状态的意义

调度器是 Runner、Model、Pre/Post Infer 之间的**共享黑板**：

```python
# Runner 写入 seed 和 latent_shape
scheduler.prepare(seed, latent_shape, ...)

# Pre_infer 读取 latents 和 timestep
x = self.scheduler.latents           # pre_infer.py:98
t = self.scheduler.timestep_input    # pre_infer.py:99

# Model.infer() 写入 noise_pred
self.scheduler.noise_pred = ...      # wan/model.py:189

# step_post 读取 noise_pred，更新 latents
model_output = self.noise_pred       # wan/scheduler.py:335
self.latents = prev_sample           # wan/scheduler.py:372
```

这种设计避免了在函数参数中传递大量中间状态。scheduler 就像一个"共享白板"，各模块在上面读写，互不耦合。

## 8. 思考题

1. **WanScheduler 的 `convert_model_output` 做了什么数学变换？为什么需要它？**
   - 提示：看公式 `x0_pred = sample - sigma_t * model_output`，对比 Euler 方法不需要这个转换

    没懂

   **参考答案**：

   核心在于两种模型的**预测目标不同**：

   ```
   Wan（扩散模型）：模型预测的是"噪声 ε"
     带噪样本 xₜ = αₜ·x₀ + σₜ·ε
     已知 xₜ 和预测的 ε，反推干净样本：x₀ = xₜ - σₜ·ε
     这就是 convert_model_output 做的事：
       x0_pred = sample - sigma_t * model_output

   HunyuanVideo（Flow Matching）：模型预测的是"速度 v"
     xₜ = (1-t)·x₀ + t·ε
     速度 v = dx/dt = ε - x₀（从数据指向噪声的方向）
     更新规则直接用：x_{t+dt} = xₜ + v·dt
     不需要先转换成 x₀
   ```

   **为什么 Wan 需要转换？** 因为 UniPC 的 Predictor/Corrector 算法内部操作的是 `x₀` 的估计值（`model_output_convert`），而不是原始的噪声预测。它需要比较不同步骤对 `x₀` 的估计来做高阶插值。所以必须先把"噪声"转换为"干净样本估计"。

   **为什么 HunyuanVideo 不需要？** Euler 方法直接用速度更新位置 `x += v·dt`，根本不需要 `x₀` 的估计，所以没有 `convert_model_output` 这步。

2. **为什么 HunyuanVideo 的 latents 用 bfloat16 而 Wan 用 float32？**
   - 提示：考虑 UniPC 多步方法中的数值稳定性需求

    我猜测UniPC对数值精度要求更高，但我不知道具体原因是啥

   **批改**：方向正确 ✅，补充具体原因。

   UniPC 对精度要求更高的具体原因有三个：

   **① 对数运算**：UniPC 的 step_post 中大量使用 `torch.log()` 计算 lambda 值：
   ```python
   # wan/scheduler.py:168-169
   lambda_t = torch.log(alpha_t) - torch.log(sigma_t)
   lambda_s0 = torch.log(alpha_s0) - torch.log(sigma_s0)
   ```
   当 `alpha_t` 接近 1 或 `sigma_t` 接近 0 时（去噪后期），`log` 的参数接近 0，BF16 的精度（尾数仅 7 位）会导致严重的精度损失。FP32（尾数 23 位）在这种情况下稳定得多。

   **② 高阶多项式**：UniPC 做二阶插值，涉及 `torch.pow(rks, i-1)` 和矩阵求解 `torch.linalg.solve(R, b)`。这些运算对输入精度非常敏感，BF16 的舍入误差会被放大。

   **③ 历史状态累积**：UniPC 保存 `model_outputs` 和 `last_sample` 等历史状态，跨步骤使用。如果用 BF16，每步的舍入误差会在历史状态中累积，几十步下来偏差显著。

   **Euler 为什么能用 BF16？** 因为它只做一次加法 `x += v * dt`，没有 log、没有矩阵求解、没有历史状态，BF16 的精度对于简单加法足够了。

3. **如果要给框架添加 DDIM 采样策略，需要实现哪些方法？**
   - 提示：继承 BaseScheduler，实现 prepare / step_pre / step_post

   继承 BaseScheduler，实现 prepare / step_pre / step_post

   **批改**：框架正确 ✅，但过于简略，缺少具体实现要点。

   **完整步骤**：

   ```python
   # lightx2v/models/schedulers/wan/ddim_scheduler.py

   class WanDDIMScheduler(BaseScheduler):
       def __init__(self, config):
           super().__init__(config)
           self.num_train_timesteps = 1000
           self.noise_pred = None

       def prepare(self, seed, latent_shape, image_encoder_output=None):
           # 1. 生成初始噪声
           self.latents = torch.randn(*latent_shape, device=AI_DEVICE)

           # 2. 计算 alpha_cumprod（DDIM 的核心参数）
           betas = torch.linspace(0.0001, 0.02, self.num_train_timesteps)
           alphas = 1.0 - betas
           self.alphas_cumprod = torch.cumprod(alphas, dim=0)

           # 3. 选择推理步的 timesteps（均匀间隔跳步）
           step_ratio = self.num_train_timesteps // self.infer_steps
           self.timesteps = torch.arange(0, self.num_train_timesteps, step_ratio).flip(0)

       def step_pre(self, step_index):
           super().step_pre(step_index)
           self.timestep_input = torch.stack([self.timesteps[step_index]])

       def step_post(self):
           # DDIM 更新公式
           t = self.timesteps[self.step_index]
           t_prev = self.timesteps[self.step_index + 1] if self.step_index + 1 < len(self.timesteps) else 0

           alpha_t = self.alphas_cumprod[t]
           alpha_prev = self.alphas_cumprod[t_prev] if t_prev > 0 else torch.tensor(1.0)

           # 预测 x0
           pred_x0 = (self.latents - torch.sqrt(1 - alpha_t) * self.noise_pred) / torch.sqrt(alpha_t)

           # 计算指向 x_{t-1} 的方向
           dir_xt = torch.sqrt(1 - alpha_prev) * self.noise_pred

           # 更新 latents（DDIM 确定性采样，eta=0）
           self.latents = torch.sqrt(alpha_prev) * pred_x0 + dir_xt
   ```

   然后在对应 Runner 的 `init_scheduler()` 中选择它：
   ```python
   def init_scheduler(self):
       if self.config["scheduler_type"] == "ddim":
           self.scheduler = WanDDIMScheduler(self.config)
       else:
           self.scheduler = WanScheduler(self.config)
   ```

   **注意**：DDIM 和 UniPC 的 `prepare` 初始化方式不同（DDIM 用 `alpha_cumprod`，UniPC 用 `sigmas`），但对外接口一致，Runner 不需要改。

4. **流式 VAE 解码为什么需要 CausalConv3d 的 cache_x 机制？如果用普通 Conv3d 会怎样？**
   - 提示：考虑卷积核在时间维度的感受野

   不太懂

   **参考答案**：

   先理解问题。流式解码是把一整个视频 latent **按时间维度切成多块**，逐块解码：

   ```
   整体解码（非流式）：
     [========= 21帧 latent =========] → VAE → [===== 81帧视频 =====]

   流式解码：
     [帧0-4] → VAE → [段1]
     [帧5-9] → VAE → [段2]
     [帧10-14] → VAE → [段3]
     ...
     拼接 → [完整视频]
   ```

   **问题出在时间维度的卷积**。假设 VAE 中有一个时间维度 kernel_size=3 的卷积：

   ```
   普通 Conv3d（非因果）：
     计算 t=5 的输出需要 t=4, t=5, t=6 的输入
     → 如果流式解码切在 t=4 和 t=5 之间，第一段没有 t=5 的数据
     → 第二段没有 t=4 的数据
     → 边界处计算结果和整体解码不一样！产生伪影

   CausalConv3d（因果卷积）：
     计算 t=5 的输出只需要 t=3, t=4, t=5 的输入（只看过去，不看未来）
     → 切块时，只要把前一块的最后 2 帧（cache_x）传给下一块
     → 下一块在计算时，用 cache_x 补上 "过去" 的信息
     → 边界处计算结果和整体解码完全一致
   ```

   看代码中 `cache_x` 的使用：

   ```python
   # video_encoders/hf/wan/vae.py:41-49
   class CausalConv3d(nn.Conv3d):
       def forward(self, x, cache_x=None):
           padding = list(self._padding)
           if cache_x is not None and self._padding[4] > 0:
               # 把前一块的尾部帧拼在当前块前面
               x = torch.cat([cache_x, x], dim=2)
               # 减少时间维度的 padding（因为 cache_x 已经提供了"过去"的信息）
               padding[4] -= cache_x.shape[2]
           x = F.pad(x, padding)
           return super().forward(x)
   ```

   **如果用普通 Conv3d 会怎样？**
   - 无法流式解码（边界处结果不一致，视频会在切块处出现明显的跳变/伪影）
   - 只能一次性解码整个视频，显存峰值很高
   - 对于长视频（几百帧），可能 GPU 显存不够用
