## 0. 网络层在整体架构中的位置

```
Pipeline 层 → Runner 层 → 网络层（本课）
                │
                ├── Runner.run_segment()
                │       └── model.infer(inputs)        ← 入口
                │               │
                │               ├── pre_infer.infer()    ← 嵌入 + RoPE
                │               ├── transformer_infer.infer()  ← N 个 Block
                │               └── post_infer.infer()   ← unpatchify
                │
                └── model = WanModel / HunyuanVideo15Model / ...
```

Runner 层调用 `model.infer(inputs)`，网络层负责**一次完整的 DiT 前向传播**——接收 latent + 条件输入，输出预测噪声。

## 1. BaseTransformerModel：网络层的抽象基类

**核心文件**：`lightx2v/models/networks/base_model.py`

### 1.2 权重与推理的分离原则

BaseTransformerModel 的核心设计理念是**将权重（weights）和推理逻辑（infer）完全分离**：

```
                    BaseTransformerModel
                    ┌───────────────────────────────────┐
                    │                                   │
           权重容器（数据）              推理模块（计算）
    ┌──────────────────────┐    ┌──────────────────────┐
    │  pre_weight           │    │  pre_infer            │
    │  transformer_weights  │    │  transformer_infer    │
    │  post_weight          │    │  post_infer           │
    └──────────────────────┘    └──────────────────────┘
          存储权重张量                 定义计算逻辑
          管理 Offload               调用权重进行计算
          管理量化格式               管理注意力算子
```

**为什么要分离？**

1. **同一套权重可以搭配不同的推理策略**：比如 WanTransformerWeights 可以搭配 WanTransformerInfer（普通推理）、WanOffloadTransformerInfer（Offload 推理）、WanTransformerInferTeaCaching（TeaCache 推理）等
2. **权重格式变化不影响推理逻辑**：FP16 权重和 FP8 量化权重都存在 weights 中，infer 代码不需要改
3. **Offload 管理集中在权重侧**：weights 负责 CPU↔GPU 搬运，infer 只需要调用权重的 `.apply()` 方法

## 2. DiT 推理的理论背景

在深入代码之前，先理解 DiT 在做什么。

### 2.1 扩散模型的核心思想

扩散模型的训练和推理基于一个简单的想法：

```
训练（正向过程）：给干净数据逐步加噪声，学会"噪声长什么样"
推理（反向过程）：从纯噪声出发，逐步去噪，恢复出干净数据
```

具体到视频生成：

```
训练：
  干净视频 x₀ ──加噪──→ 带噪视频 xₜ ──输入DiT──→ 预测噪声 ε̂
                                                    ↕ 与真实噪声对比
                                              计算损失，更新权重

推理（LightX2V 做的事）：
  纯噪声 xₜ ──→ DiT 预测噪声 ──→ 去掉噪声 ──→ xₜ₋₁ ──→ ... ──→ x₀（干净视频）
            ↑                                      │
            └──── 重复 infer_steps 次 ─────────────┘
```

**这就是第3课讲的 `run_segment()` 循环在做的事**：每个 step 调用 `model.infer()`（DiT 前向传播）预测噪声，然后 `scheduler.step_post()` 根据预测结果更新 latent。

### 2.2 为什么用 Transformer 替代 UNet？

传统扩散模型（如 Stable Diffusion 1.x）用 UNet 做去噪网络。DiT（Diffusion Transformer）用 Transformer 替代了 UNet：

| 维度 | UNet | DiT (Transformer) |
|------|------|-------------------|
| 结构 | 编码器-解码器，有跳跃连接 | 纯 Transformer Block 堆叠 |
| 缩放性 | 加深/加宽受限 | 参数量线性缩放，容易做大 |
| 输入格式 | 2D/3D 特征图 | 1D token 序列（patchify 后） |
| 条件注入 | cross-attention + concat | AdaLN + cross-attention |

**Transformer 的优势**：在大规模数据上，Transformer 的 scaling law 更好——参数量翻倍，生成质量持续提升。这是 Wan（14B 参数）和 HunyuanVideo 等大模型选择 DiT 的根本原因。

### 2.3 Patchify：把视频变成 token 序列

Transformer 处理的是 1D token 序列，但视频是 3D 数据（时间×高度×宽度）。需要 **patchify** 将视频 latent 转为 token：

```
VAE latent:  [C=16, T=21, H=60, W=104]
                    │
                    │  patch_embedding（3D 卷积，stride=(1,2,2)）
                    ▼
特征图:       [dim, T'=21, H'=30, W'=52]
                    │
                    │  flatten + transpose
                    ▼
token 序列:   [seq_len=21×30×52=32760, dim]
```

每个 token 对应原始 latent 中一个 1×2×2 的 patch。这就是 `pre_infer` 中 `patch_embedding` 做的事。

推理结束后，`post_infer` 的 `unpatchify` 做逆操作——把 token 序列还原回视频形状。

### 2.4 条件注入：如何让 DiT "听话"

DiT 需要根据条件（文本描述、时间步、图像）来指导生成。LightX2V 中有三种条件注入机制：

**① AdaLN（Adaptive Layer Norm）— 注入时间步信息**

普通 LayerNorm 的公式：`y = (x - mean) / std * γ + β`，其中 γ 和 β 是可学习参数。

AdaLN 把 γ 和 β 替换为**从 timestep 动态生成的参数**：

```python
# 对应代码：pre_infer 中的 time_embedding → embed0
# embed0 = [shift, scale, gate, c_shift, c_scale, c_gate]

# transformer_infer 中的 pre_process:
shift_msa, scale_msa, gate_msa, ... = (modulation + embed0).chunk(6, dim=1)

# 应用到 LayerNorm 后的特征：
y = LayerNorm(x) * (1 + scale) + shift    # scale 和 shift 来自 timestep
output = residual + y * gate               # gate 控制残差强度
```

**为什么用 AdaLN 而不是直接拼接 timestep？** 因为 timestep 影响的是特征的统计分布（不同去噪阶段特征的均值和方差不同），AdaLN 直接调整分布比拼接后让网络自己学更高效。

**② Cross-Attention — 注入文本/图像条件**

```
Q = 视频 token      ← 来自当前 Block 的特征
K, V = 文本 token   ← 来自 T5/Qwen2.5-VL 编码的文本特征

Attention(Q, K, V) = softmax(QK^T / √d) V
```

视频 token 通过 attention 机制"查询"文本特征中的相关信息。比如文本说"红色的猫"，与"猫"相关的视频区域会获得更高的 attention 权重。

对应代码中 `infer_cross_attn()` 的逻辑。Wan 模型有两个 cross-attention：
- `cross_attn_1`：与 T5 文本特征交互
- `cross_attn_2`：与 CLIP 图像特征交互（I2V 任务）

**③ Concatenation — 注入 VAE 图像 latent（I2V）**

```python
# pre_infer.py:124-125
# 直接把 VAE 编码的图像 latent 拼接到噪声 latent 的通道维
x = torch.cat([x, image_encoder], dim=0)  # 沿通道维拼接
```

这是最直接的方式——把首帧信息"硬编码"进输入。

### 2.5 Self-Attention + RoPE：视频 token 之间的交互

Self-Attention 让每个视频 token 都能看到所有其他 token，从而建立全局的时空关系。

但标准 Transformer 没有位置信息——打乱 token 顺序结果不变。**RoPE（Rotary Position Embedding）** 通过旋转 Q 和 K 来编码位置：

```python
# transformer_infer.py:193
q, k = self.apply_rope_func(q, k, cos_sin)
```

Wan 使用 **3D RoPE**——分别对时间、高度、宽度三个维度编码位置，让模型知道每个 token 在视频中的时空坐标。

### 2.6 CFG（Classifier-Free Guidance）：增强条件控制

CFG 的核心公式：

```
noise_pred = noise_uncond + scale * (noise_cond - noise_uncond)
```

- `noise_cond`：有文本条件时的预测（"一只红色的猫"）
- `noise_uncond`：无文本条件时的预测（空 prompt）
- `scale`：引导强度（通常 5~7）

直觉理解：`noise_cond - noise_uncond` 表示"文本条件带来的方向"，`scale` 放大这个方向，让生成结果更贴合文本描述。scale 越大越贴合但可能过饱和/不自然。

**对应代码中的实现**：

```python
# wan/model.py:186-189
noise_pred_cond = self._infer_cond_uncond(inputs, infer_condition=True)   # 有条件
noise_pred_uncond = self._infer_cond_uncond(inputs, infer_condition=False) # 无条件
self.scheduler.noise_pred = noise_pred_uncond + scale * (noise_pred_cond - noise_pred_uncond)
```

这就是为什么 `infer()` 中启用 CFG 时 DiT 要跑**两次**前向传播——一次有条件，一次无条件。

### 2.7 一个 step 的完整数据流

把以上理论串起来，对应代码中的一个 step：

```
scheduler.step_pre()
    └── 准备 timestep_input（当前噪声级别）

model.infer(inputs)
    │
    ├── [有 CFG] _infer_cond_uncond(infer_condition=True)   ← 条件推理
    │       │
    │       ├── pre_infer.infer()
    │       │     ├── latent + image_latent → patchify → token 序列  [Patchify]
    │       │     ├── timestep → time_embedding → embed0             [AdaLN 参数]
    │       │     ├── text → text_embedding → context                [Cross-Attn 条件]
    │       │     └── 3D RoPE → cos_sin                              [位置编码]
    │       │
    │       ├── transformer_infer.infer()
    │       │     └── for block in 48 blocks:
    │       │           ├── AdaLN(x, embed0)                         [时间步调制]
    │       │           ├── SelfAttn(x, RoPE)                        [视频内部交互]
    │       │           ├── CrossAttn(x, context)                    [与文本交互]
    │       │           ├── CrossAttn(x, clip_fea)                   [与图像交互]
    │       │           └── FFN(x) + 残差                            [非线性变换]
    │       │
    │       └── post_infer.infer()
    │             └── unpatchify → noise_pred_cond                   [还原视频形状]
    │
    ├── [有 CFG] _infer_cond_uncond(infer_condition=False)  ← 无条件推理
    │       └── 同上，但 context 用 context_null
    │           → noise_pred_uncond
    │
    └── CFG 合并：
        noise_pred = uncond + scale * (cond - uncond)

scheduler.step_post()
    └── 用 noise_pred 更新 latents（去掉一步噪声）
```

## 3. 三阶段推理架构

### 2.1 _infer_cond_uncond()：一次前向传播

```python
# wan/model.py:102-123
@torch.no_grad()
def _infer_cond_uncond(self, inputs, infer_condition=True):
    self.scheduler.infer_condition = infer_condition

    # 阶段1：pre_infer — 嵌入 + 位置编码
    pre_infer_out = self.pre_infer.infer(self.pre_weight, inputs)

    # 序列并行：切分序列
    if self.config["seq_parallel"]:
        pre_infer_out = self._seq_parallel_pre_process(pre_infer_out)

    # 阶段2：transformer_infer — N 个 Block 的主体推理
    x = self.transformer_infer.infer(self.transformer_weights, pre_infer_out)

    # 序列并行：聚合序列
    if self.config["seq_parallel"]:
        x = self._seq_parallel_post_process(x)

    # 阶段3：post_infer — unpatchify + 输出
    noise_pred = self.post_infer.infer(x, pre_infer_out)[0]

    return noise_pred
```

**三阶段的数据流**：

```
inputs (latents + text + image 条件)
    │
    ▼
┌────────────────────────────────────────────────────┐
│ pre_infer（WanPreInfer）                             │
│                                                      │
│  latents → patch_embedding → x  (序列化)             │
│  timestep → time_embedding → embed0 (时间嵌入)       │
│  text → text_embedding → context (文本特征)          │
│  clip_fea → img_emb → context (拼接到文本特征)       │

│                                                      │

│    embed0:   [1, 6, dim]         调制参数             │

│    cos_sin:  [seq_len, head_dim] 位置编码             │
└──────────────────────┬─────────────────────────────┘
                       │

┌────────────────────────────────────────────────────┐

```

### 2.2 infer()：CFG 的处理

`_infer_cond_uncond` 只做一次前向传播。外层的 `infer()` 负责 CFG（Classifier-Free Guidance）逻辑：

```python
# wan/model.py:159-199
@torch.no_grad()
def infer(self, inputs):
    # Offload：推理前把权重搬到 GPU
    if self.cpu_offload:
        if self.offload_granularity != "model":
            self.pre_weight.to_cuda()
            self.transformer_weights.non_block_weights_to_cuda()

    if self.config["enable_cfg"]:
        if self.config["cfg_parallel"]:
            # CFG 并行：两张 GPU 各跑一个条件
            if cfg_p_rank == 0:
                noise_pred = self._infer_cond_uncond(inputs, infer_condition=True)
            else:
                noise_pred = self._infer_cond_uncond(inputs, infer_condition=False)
            # all_gather 交换结果
            dist.all_gather(noise_pred_list, noise_pred, group=cfg_p_group)
            noise_pred_cond = noise_pred_list[0]
            noise_pred_uncond = noise_pred_list[1]
        else:
            # 非并行：同一张 GPU 跑两次
            noise_pred_cond = self._infer_cond_uncond(inputs, infer_condition=True)
            noise_pred_uncond = self._infer_cond_uncond(inputs, infer_condition=False)

        # CFG 公式：pred = uncond + scale * (cond - uncond)
        self.scheduler.noise_pred = noise_pred_uncond + \
            self.scheduler.sample_guide_scale * (noise_pred_cond - noise_pred_uncond)
    else:
        # 无 CFG：只跑一次
        self.scheduler.noise_pred = self._infer_cond_uncond(inputs, infer_condition=True)

    # Offload：推理后把权重搬回 CPU
    if self.cpu_offload:
        if self.offload_granularity != "model":
            self.pre_weight.to_cpu()
            self.transformer_weights.non_block_weights_to_cpu()
```

**CFG 的成本**：启用 CFG 时 DiT 要跑两次前向传播（cond + uncond），计算量翻倍。CFG 并行可以将这两次分到两张 GPU 上，恢复到接近单次的延迟。

## 4. pre_infer：嵌入与位置编码

**核心文件**：`lightx2v/models/networks/wan/infer/pre_infer.py`

### 3.1 WanPreInfer.infer() 做了什么

```python
# wan/infer/pre_infer.py:96-216
@torch.no_grad()
def infer(self, weights, inputs, kv_start=0, kv_end=0):
    # 1. 获取当前 latent 和 timestep
    x = self.scheduler.latents          # [C, T, H, W]
    t = self.scheduler.timestep_input   # 当前时间步

    # 2. 获取文本条件
    if self.scheduler.infer_condition:
        context = inputs["text_encoder_output"]["context"]      # 正样本
    else:
        context = inputs["text_encoder_output"]["context_null"]  # 负样本

    # 3. I2V 任务：拼接 VAE 编码的图像 latent
    if self.task in ["i2v", "flf2v", ...]:
        image_encoder = inputs["image_encoder_output"]["vae_encoder_out"]
        if image_encoder is not None:
            x = torch.cat([x, image_encoder], dim=0)  # 沿通道维拼接

    # 4. Patch Embedding：3D 卷积，将视频 latent 转为 token 序列
    x = weights.patch_embedding.apply(x.unsqueeze(0))
    #   输入：[1, C_in, T, H, W]
    #   输出：[1, C_out, T', H', W']  （stride=(1,2,2) 空间下采样2倍）
    grid_sizes_t, grid_sizes_h, grid_sizes_w = x.shape[2:]
    x = x.flatten(2).transpose(1, 2).contiguous()
    #   reshape 为：[1, seq_len, dim]  其中 seq_len = T' * H' * W'

    # 5. Time Embedding：时间步 → 嵌入向量
    embed = sinusoidal_embedding_1d(self.freq_dim, t.flatten())
    embed = weights.time_embedding_0.apply(embed)
    embed = torch.nn.functional.silu(embed)
    embed = weights.time_embedding_2.apply(embed)
    embed0 = torch.nn.functional.silu(embed)
    embed0 = weights.time_projection_1.apply(embed0).unflatten(1, (6, self.dim))
    #   输出：[1, 6, dim]  — 6 个调制参数

    # 6. Text Embedding：文本特征投影
    out = weights.text_embedding_0.apply(context.squeeze(0))
    out = torch.nn.functional.gelu(out, approximate="tanh")
    context = weights.text_embedding_2.apply(out)

    # 7. CLIP 特征：投影并拼接到 context
    if self.task in ["i2v", ...] and self.config.get("use_image_encoder", True):
        context_clip = weights.proj_0.apply(clip_fea)
        context_clip = weights.proj_1.apply(context_clip)
        context_clip = torch.nn.functional.gelu(context_clip, approximate="none")
        context_clip = weights.proj_3.apply(context_clip)
        context_clip = weights.proj_4.apply(context_clip)
        context = torch.concat([context_clip, context], dim=0)
        # CLIP 特征拼在 text context 前面

    # 8. RoPE 位置编码
    if self.cos_sin is None or self.grid_sizes != grid_sizes.tuple:
        self.cos_sin = self.prepare_cos_sin(grid_sizes.tuple, self.freqs.clone())

    # 9. 返回打包的输出
    return WanPreInferModuleOutput(
        embed=embed,
        grid_sizes=grid_sizes,
        x=x.squeeze(0),        # [seq_len, dim]
        embed0=embed0.squeeze(0),  # [6, dim] — 6 个调制参数
        context=context,         # [text_len, dim]
        cos_sin=self.cos_sin,    # RoPE
    )
```

**embed0 的 6 个调制参数**：

DiT 使用 AdaLN（Adaptive Layer Norm）机制，每个 Block 需要 6 个调制参数：

```
embed0 = [shift_msa, scale_msa, gate_msa, c_shift_msa, c_scale_msa, c_gate_msa]
           │          │          │          │             │             │
           ▼          ▼          ▼          ▼             ▼             ▼
        SelfAttn   SelfAttn   SelfAttn   FFN 分支     FFN 分支      FFN 分支
        的 shift   的 scale   的 gate    的 shift     的 scale      的 gate
```

- **shift/scale**：控制 LayerNorm 后的仿射变换（让模型根据 timestep 调整特征分布）
- **gate**：控制残差连接的强度（让模型根据 timestep 决定每个分支的贡献权重）

### 3.2 RoPE 位置编码

Wan 模型的 RoPE 是 3D 的——分别对时间、高度、宽度三个维度做位置编码：

```python
# wan/infer/pre_infer.py:55-94
def prepare_cos_sin(self, grid_sizes, freqs):
    f, h, w = grid_sizes
    # freqs 分成三段，分别对应时间、高度、宽度
    freqs = freqs.split([c - 2*(c//3), c//3, c//3], dim=1)
    cos_sin = torch.cat([
        freqs[0][:f].view(f, 1, 1, -1).expand(f, h, w, -1),  # 时间维
        freqs[1][:h].view(1, h, 1, -1).expand(f, h, w, -1),  # 高度维
        freqs[2][:w].view(1, 1, w, -1).expand(f, h, w, -1),  # 宽度维
    ], dim=-1)
    cos_sin = cos_sin.reshape(seq_len, -1)  # [seq_len, head_dim]
```

## 5. transformer_infer：主体推理循环

**核心文件**：`lightx2v/models/networks/wan/infer/transformer_infer.py`

### 4.1 整体结构

```python
# wan/infer/transformer_infer.py:92-96
@torch.no_grad()
def infer(self, weights, pre_infer_out):
    self.cos_sin = pre_infer_out.cos_sin
    self.reset_infer_states()
    # 1. 遍历所有 Block
    x = self.infer_main_blocks(weights.blocks, pre_infer_out)
    # 2. 最终的 norm + head
    return self.infer_non_blocks(weights, x, pre_infer_out.embed)
```

### 4.2 infer_block()：单个 Block 的推理

```python
# wan/infer/transformer_infer.py:132-155
def infer_block(self, block, x, pre_infer_out):
    # 1. 调制参数预处理
    shift_msa, scale_msa, gate_msa, c_shift_msa, c_scale_msa, c_gate_msa = \
        self.pre_process(block.compute_phases[0].modulation, pre_infer_out.embed0)

    # 2. Self-Attention
    y_out = self.infer_self_attn(block.compute_phases[0], x, shift_msa, scale_msa)

    # 3. Cross-Attention（与文本条件交互）
    x, attn_out = self.infer_cross_attn(
        block.compute_phases[1], x, pre_infer_out.context, y_out, gate_msa
    )

    # 4. FFN
    y = self.infer_ffn(block.compute_phases[2], x, attn_out, c_shift_msa, c_scale_msa)

    # 5. 残差连接
    x = self.post_process(x, y, c_gate_msa, pre_infer_out)

    return x
```

**注意 `block.compute_phases[0/1/2]` 的用法**：这里可以清楚地看到，推理代码（infer）通过索引访问权重容器（weights）的 phases。推理逻辑不关心权重存在哪里（GPU、CPU、还是磁盘），只调用 `.apply()` 方法。

### 4.3 infer_self_attn()：自注意力

```python
# wan/infer/transformer_infer.py:171-244
def infer_self_attn(self, phase, x, shift_msa, scale_msa):
    # 1. AdaLN：LayerNorm + 调制
    norm1_out = phase.norm1.apply(x)
    norm1_out = self.modulate_func(norm1_out, scale=scale_msa, shift=shift_msa)

    # 2. QKV 投影
    q = phase.self_attn_norm_q.apply(phase.self_attn_q.apply(norm1_out)).view(s, n, d)
    k = phase.self_attn_norm_k.apply(phase.self_attn_k.apply(norm1_out)).view(s, n, d)
    v = phase.self_attn_v.apply(norm1_out).view(s, n, d)

    # 3. RoPE
    q, k = self.apply_rope_func(q, k, cos_sin)

    # 4. Attention 计算（通过注册器选择算子）
    attn_out = phase.self_attn_1.apply(q=q, k=k, v=v, ...)

    # 5. Output 投影
    y = phase.self_attn_o.apply(attn_out)
    return y
```

**`phase.self_attn_1.apply()` 的背后**：这不是普通的矩阵乘法，而是通过 `ATTN_WEIGHT_REGISTER` 注册的注意力算子。根据 `attn_mode` 配置，它可能是 Flash Attention 2、Flash Attention 3、Sage Attention 等。

### 4.4 infer_cross_attn()：交叉注意力

```python
# wan/infer/transformer_infer.py:246-319
def infer_cross_attn(self, phase, x, context, y_out, gate_msa):
    # 1. 残差连接（Self-Attention 输出）
    x = x + y_out * gate_msa

    # 2. LayerNorm
    norm3_out = phase.norm3.apply(x)

    # 3. 分离 CLIP context 和 Text context
    if self.task in ["i2v", ...] and self.config.get("use_image_encoder", True):
        context_img = context[:257]   # CLIP 特征（前 257 个 token）
        context = context[257:]       # T5 文本特征

    # 4. Text Cross-Attention
    q = phase.cross_attn_q.apply(norm3_out)
    k = phase.cross_attn_k.apply(context)
    v = phase.cross_attn_v.apply(context)
    attn_out = phase.cross_attn_1.apply(q=q, k=k, v=v, ...)

    # 5. Image Cross-Attention（如果有 CLIP）
    if context_img is not None:
        k_img = phase.cross_attn_k_img.apply(context_img)
        v_img = phase.cross_attn_v_img.apply(context_img)
        img_attn_out = phase.cross_attn_2.apply(q=q, k=k_img, v=v_img, ...)
        attn_out.add_(img_attn_out)  # 两个 cross-attn 结果相加

    # 6. Output 投影
    attn_out = phase.cross_attn_o.apply(attn_out)
    return x, attn_out
```

**Wan 的 Cross-Attention 有两个**：一个与 T5 文本特征交互（`cross_attn_1`），一个与 CLIP 图像特征交互（`cross_attn_2`）。结果直接相加。

### 4.5 infer_non_blocks()：最终输出

```python
# wan/infer/transformer_infer.py:102-124
def infer_non_blocks(self, weights, x, e):
    # 1. 最终调制
    modulation = weights.head_modulation.tensor
    e = (modulation + e.unsqueeze(1)).chunk(2, dim=1)  # shift 和 scale

    # 2. 最终 LayerNorm
    x = weights.norm.apply(x)

    # 3. 调制
    x = x * (1 + e[1]) + e[0]

    # 4. 线性投影到 output 维度
    x = weights.head.apply(x)  # [seq_len, dim] → [seq_len, out_dim]

    return x
```

## 6. post_infer：unpatchify

**核心文件**：`lightx2v/models/networks/wan/infer/post_infer.py`

post_infer 是三个阶段中最简单的——它把一维 token 序列重新 reshape 回视频形状：

```python
# wan/infer/post_infer.py:8-31
class WanPostInfer:
    def __init__(self, config):
        self.out_dim = config["out_dim"]
        self.patch_size = (1, 2, 2)

    @torch.no_grad()
    def infer(self, x, pre_infer_out):
        x = self.unpatchify(x, pre_infer_out.grid_sizes.tuple)
        return [u.float() for u in x]

    def unpatchify(self, x, grid_sizes):
        c = self.out_dim
        # [seq_len, out_dim] → [T', H', W', 1, 2, 2, C]
        x = x[:math.prod(grid_sizes)].view(*grid_sizes, *self.patch_size, c)
        # [T', H', W', 1, 2, 2, C] → [C, T'*1, H'*2, W'*2] = [C, T, H, W]
        x = torch.einsum("fhwpqrc->cfphqwr", x)
        x = x.reshape(c, *[i * j for i, j in zip(grid_sizes, self.patch_size)])
        return [x]
```

**unpatchify 的作用**：pre_infer 中 patch_embedding 用 stride=(1,2,2) 下采样，unpatchify 是其逆操作——把每个 token 还原为 1×2×2 的 patch，恢复空间分辨率。

### 6.2 注册器驱动的权重类型

权重容器不直接创建 `nn.Linear` 等 PyTorch 模块，而是通过注册器选择具体的权重实现：

```python
# wan/weights/pre_weights.py:13-21
self.add_module(
```

**注册器的种类**：

| 注册器 | 用途 | 常见实现 |
|--------|------|---------|
| `MM_WEIGHT_REGISTER` | 矩阵乘法（Linear） | Default, FP8, INT8, NVFP4 |
| `ATTN_WEIGHT_REGISTER` | 注意力算子 | flash_attn2, flash_attn3, sage_attn |
| `LN_WEIGHT_REGISTER` | LayerNorm | torch（标准实现） |
| `RMS_WEIGHT_REGISTER` | RMSNorm | torch, triton |
| `CONV3D_WEIGHT_REGISTER` | 3D 卷积 | Default |
| `TENSOR_REGISTER` | 纯张量（不做计算） | Default |

### 7.3 注册器驱动的权重系统详解

第1课介绍了 `RUNNER_REGISTER` 的注册机制（名字→类的映射）。权重注册器（`MM_WEIGHT_REGISTER`、`ATTN_WEIGHT_REGISTER` 等）的机制完全一样，但用途不同——它让**同一份权重定义代码**可以根据配置选择不同的计算实现。

**核心问题**：同一个 `self_attn_q`（Self-Attention 的 Q 投影），在不同配置下需要不同的计算方式。如何避免 if-else 泛滥？

**方案：通过注册器，用配置字符串选择实现类**。

#### 装饰器到底做了什么？逐步拆解

先看一个最小的例子，把"装饰器"翻译成等价的普通 Python 代码：

```python
# ===== 第0步：创建注册器（一个花哨的字典）=====
# registry_factory.py:59
MM_WEIGHT_REGISTER = Register()
# 此时它是空的：MM_WEIGHT_REGISTER._dict = {}
```

Register 类的完整实现（`lightx2v/utils/registry_factory.py:4-51`）：

```python
class Register(dict):
    def __init__(self, *args, **kwargs):
        super(Register, self).__init__(*args, **kwargs)
        self._dict = {}       # 真正存数据的字典

    def __call__(self, target_or_name):
        """让实例可以当装饰器用"""
        if callable(target_or_name):
            # @REGISTER         ← 不带参数，target_or_name 就是类本身
            return self.register(target_or_name)
        else:
            # @REGISTER("xxx")  ← 带参数，target_or_name 是字符串 "xxx"
            return lambda x: self.register(x, key=target_or_name)

    def register(self, target, key=None):
        if not callable(target):
            raise Exception(f"Error: {target} must be callable!")
        if key is None:
            key = target.__name__    # 不指定名字就用类名
        if key in self._dict:
            raise Exception(f"{key} already exists.")
        self[key] = target           # 存入字典
        return target                # 返回原类（不做修改）

    def __setitem__(self, key, value):
        self._dict[key] = value

    def __getitem__(self, key):
        return self._dict[key]       # REGISTER["xxx"] 从字典取出类
```

所有注册器都是这个类的实例（`registry_factory.py:59-71`）：

```python
MM_WEIGHT_REGISTER = Register()           # 矩阵乘法权重
ATTN_WEIGHT_REGISTER = Register()         # 注意力算子
RMS_WEIGHT_REGISTER = Register()          # RMSNorm 权重
LN_WEIGHT_REGISTER = Register()           # LayerNorm 权重
CONV3D_WEIGHT_REGISTER = Register()       # 3D 卷积
TENSOR_REGISTER = Register()              # 纯张量
RUNNER_REGISTER = Register()              # Runner（第1课讲的）
# ... 还有 ROPE_REGISTER, SPARSE_MASK_GENERATOR_REGISTER 等
```

理解了 `Register` 类之后，下面的装饰器流程就很清楚了：

```python
# ===== 第1步：用装饰器注册一个类 =====
# mm_weight.py:263-264
@MM_WEIGHT_REGISTER("Default")
class MMWeight(MMWeightTemplate):
    def apply(self, input_tensor):
        return torch.mm(input_tensor, self.weight)
```

这个 `@装饰器` 看起来很魔法，但 Python 会把它翻译成以下两步普通代码：

```python
# Python 实际执行的等价代码：

# 步骤 A：调用 MM_WEIGHT_REGISTER("Default")
#          参数 "Default" 是字符串，不是 callable
#          所以走 __call__ 的 else 分支，返回一个 lambda
temp_decorator = MM_WEIGHT_REGISTER("Default")
# 此时 temp_decorator = lambda x: self.register(x, key="Default")

# 步骤 B：用这个 lambda 去装饰 MMWeight 类
MMWeight = temp_decorator(MMWeight)
# 内部调用 self.register(MMWeight, key="Default")
# 也就是：self._dict["Default"] = MMWeight
# 然后返回 MMWeight 本身（类没有被修改）
```

**执行完毕后**：`MM_WEIGHT_REGISTER._dict["Default"] = MMWeight`

就这么简单——**装饰器的唯一副作用就是往字典里存了一条记录**。类本身没有任何改变。

```python
# ===== 第2步：同理注册其他实现 =====

class MMWeightFP8Vllm(MMWeightQuantTemplate):
    def apply(self, input_tensor):
        return vllm_ops.cutlass_scaled_mm(...)  # FP8 GEMM

@MM_WEIGHT_REGISTER("fp8-sgl")
class MMWeightFP8Sgl(MMWeightQuantTemplate):
    def apply(self, input_tensor):
        return sgl_kernel.fp8_scaled_mm(...)    # SGL FP8 GEMM

# 此时字典里有：
# MM_WEIGHT_REGISTER._dict = {
#     "Default":   MMWeight,
#     "fp8-vllm":  MMWeightFP8Vllm,
#     "fp8-sgl":   MMWeightFP8Sgl,
# }
```

```python

# wan/weights/transformer_weights.py:265
mm_type = config["dit_quant_scheme"]  # 比如 "fp8-vllm"

self.self_attn_q = MM_WEIGHT_REGISTER[mm_type](    # 取出类
    "blocks.0.self_attn.q.weight",                   # 用参数实例化
    "blocks.0.self_attn.q.bias",

# 等价于：
# cls = MM_WEIGHT_REGISTER._dict["fp8-vllm"]   → MMWeightFP8Vllm
# self.self_attn_q = cls("blocks.0.self_attn.q.weight", ...)
```

```python

# 如果 mm_type 是 "Default"，这里调用的是 MMWeight.apply()        → torch.mm
# 如果 mm_type 是 "fp8-vllm"，这里调用的是 MMWeightFP8Vllm.apply() → vllm FP8 GEMM
```

**总结：整个机制就三句话**：
1. 装饰器把"名字→类"的映射存进字典
2. 权重容器创建时，用配置中的名字从字典里取出类来实例化
3. 推理时调用实例的 `.apply()`，自动走到正确的实现

#### 那这些装饰器什么时候执行？

装饰器在 **`import` 时**执行。看 `__init__.py` 的 import 链：

```
lightx2v/common/ops/__init__.py
    from .mm import *          ← import mm 包
        ↓
lightx2v/common/ops/mm/__init__.py
    from .mm_weight import *   ← import mm_weight.py
        ↓
lightx2v/common/ops/mm/mm_weight.py
    @MM_WEIGHT_REGISTER("Default")    ← import 时装饰器执行
    class MMWeight: ...               ← "Default" → MMWeight 存入字典

    @MM_WEIGHT_REGISTER("fp8-vllm")   ← import 时装饰器执行
    class MMWeightFP8Vllm: ...        ← "fp8-vllm" → MMWeightFP8Vllm 存入字典
```

所以当用户写 `from lightx2v import LightX2VPipeline` 时，所有 `__init__.py` 会级联导入，所有装饰器都执行了，注册器字典就填满了。**后续使用时只是查字典，不需要任何额外操作**。

#### 第一层：模板类定义接口

```python
# common/ops/mm/mm_weight.py:110-260
class MMWeightTemplate(metaclass=ABCMeta):
    """所有矩阵乘法权重的抽象基类"""
    def __init__(self, weight_name, bias_name, ...):
        self.weight_name = weight_name   # 如 "blocks.0.self_attn.q.weight"
        self.bias_name = bias_name       # 如 "blocks.0.self_attn.q.bias"

    @abstractmethod
    def load(self, weight_dict):
        """从权重字典中加载张量"""
        pass

    @abstractmethod
    def apply(self, input_tensor):
        """执行计算（矩阵乘法）"""
        pass

    # 还定义了 LoRA、Offload、state_dict 等通用能力
    def apply_lora(self, input_tensor):
        h = torch.mm(input_tensor, self.lora_down.t())
        out = torch.mm(h, self.lora_up.t())
        return self.lora_strength * self.lora_scale * out
```

注意力算子也有类似的模板：

```python
# common/ops/attn/template.py:4-15
class AttnWeightTemplate(metaclass=ABCMeta):
    """所有注意力算子的抽象基类"""
    def load(self, weight_dict):
        pass   # 注意力算子没有权重（不像 Linear）

    @abstractmethod
    def apply(self, input_tensor):
        """执行注意力计算"""
        pass
```

#### 第二层：具体实现通过装饰器注册

```python
# Default：普通 BF16 矩阵乘法
@MM_WEIGHT_REGISTER("Default")
class MMWeight(MMWeightTemplate):
    def load(self, weight_dict):
        self.weight = weight_dict[self.weight_name].t()  # 转置存储
        self.bias = weight_dict.get(self.bias_name, None)

    def apply(self, input_tensor):
        if not self.has_lora_branch:
            return torch.addmm(self.bias, input_tensor, self.weight)
        else:
            # 有 LoRA：普通矩阵乘 + LoRA 分支
            return torch.addmm(self.bias, input_tensor, self.weight) + self.apply_lora(input_tensor)

# FP8 量化：权重和激活都用 FP8 计算
@MM_WEIGHT_REGISTER("fp8-vllm")
class MMWeightFP8Vllm(MMWeightQuantTemplate):
    def load(self, weight_dict):
        self.weight = weight_dict[self.weight_name]  # 已是 FP8 格式
        self.weight_scale = weight_dict[self.weight_name + "_scale"]

    def apply(self, input_tensor):
        # 激活量化为 FP8
        input_quant, input_scale = self.act_quant_func(input_tensor)
        # 调用 vLLM 的 FP8 GEMM kernel
        return vllm_ops.cutlass_scaled_mm(input_quant, self.weight, input_scale, self.weight_scale, ...)

# Flash Attention 2
@ATTN_WEIGHT_REGISTER("flash_attn2")
class FlashAttn2Weight(AttnWeightTemplate):
    def apply(self, q, k, v, cu_seqlens_q, cu_seqlens_kv, ...):
        return flash_attn_varlen_func(q, k, v, cu_seqlens_q, cu_seqlens_kv, ...)

# Sage Attention（更省显存的注意力）
@ATTN_WEIGHT_REGISTER("sage_attn")
class SageAttnWeight(AttnWeightTemplate):
    def apply(self, q, k, v, ...):
        return sageattn(q, k, v, ...)
```

#### 完整的数据流

从配置到执行的完整路径：

```
用户配置:  dit_quant_scheme = "fp8-vllm"
           self_attn_1_type = "flash_attn2"
                    │
                    ▼
权重容器创建（WanSelfAttention.__init__）:
    self.self_attn_q = MM_WEIGHT_REGISTER["fp8-vllm"](...)  → MMWeightFP8Vllm 实例
    self.self_attn_1 = ATTN_WEIGHT_REGISTER["flash_attn2"]() → FlashAttn2Weight 实例
                    │
                    ▼
权重加载（_apply_weights）:
    self.self_attn_q.load(weight_dict)  → 从字典取出 FP8 权重 + scale
    self.self_attn_1.load(weight_dict)  → 注意力无权重，跳过
                    │
                    ▼
推理执行（transformer_infer.infer_self_attn）:
    q = phase.self_attn_q.apply(norm1_out)       → FP8 量化 GEMM
    attn_out = phase.self_attn_1.apply(q, k, v)  → Flash Attention 2
```

**推理代码中只看到 `.apply()` 调用**，完全不知道底层是 BF16 还是 FP8，是 Flash Attention 还是 Sage Attention。这就是注册器+模板方法模式的威力。

#### 与第1课 RUNNER_REGISTER 的类比

| 维度 | RUNNER_REGISTER | MM_WEIGHT_REGISTER |
|------|----------------|-------------------|
| 作用 | 根据 model_cls 选择 Runner | 根据 quant_scheme 选择矩阵乘法 |
| 注册时机 | import 时（装饰器执行） | import 时（装饰器执行） |
| 使用时机 | `_init_runner()` 中 | 权重容器 `__init__` 中 |
| 选择依据 | `config["model_cls"]` | `config["dit_quant_scheme"]` |
| 统一接口 | `runner.run_pipeline()` | `weight.apply(x)` |

**所有注册器的底层实现都是第1课讲的 `Register` 类**——一个可以当装饰器用的字典。

### 7.4 .apply() 方法的多态行为

**所有权重对外只暴露 `.apply()` 方法**。推理代码调用 `weight.apply(x)` 时，内部可能是：
- 普通的 `torch.mm(x, weight)`（Default）
- FP8 量化 GEMM（fp8-vllm / fp8-sgl）
- INT8 量化 GEMM（int8-torchao）
- GGUF 反量化后计算（gguf-Q4_K_M）
- 带 LoRA 的 `torch.mm(x, weight) + lora_strength * x @ lora_down @ lora_up`

推理代码完全不需要关心这些差异。

### 6.3 权重加载流程

```python
# base_model.py:205-230
def _apply_weights(self, weight_dict=None):
    # 把权重字典分发到各容器
    self.pre_weight.load(self.original_weight_dict)
    self.transformer_weights.load(self.original_weight_dict)
    if hasattr(self, "post_weight"):
        self.post_weight.load(self.original_weight_dict)

    # 如果有 LoRA，注册 LoRA 权重
    if self.config.get("lora_dynamic_apply", False):
        self._register_lora(self.lora_path, self.lora_strength)

    del self.original_weight_dict
    torch.cuda.empty_cache()
```

每个权重模块的 `load()` 方法会根据自己的 key（如 `"text_embedding.0.weight"`）从字典中提取对应的张量。

### 6.4 Offload 缓冲区

当启用 Offload 时，`WanTransformerWeights` 还会创建额外的 GPU/CPU 缓冲区：

```python
# wan/weights/transformer_weights.py:55-100
def register_offload_buffers(self, config, lazy_load_path, lora_path):
    if config["cpu_offload"]:
        if config["offload_granularity"] == "block":
            # 双缓冲：GPU 上放 2 个 Block 大小的缓冲区
            self.offload_block_cuda_buffers = WeightModuleList([
                WanTransformerAttentionBlock(..., create_cuda_buffer=True)
                for i in range(2)  # 双缓冲
            ])
        elif config["offload_granularity"] == "phase":
            # 更细粒度：GPU 上只放 phase 级别的缓冲区
            self.offload_phase_cuda_buffers = WanTransformerAttentionBlock(
                ..., create_cuda_buffer=True
            ).compute_phases  # 只取 phases 部分
```

双缓冲的工作方式：当 GPU 在用 buffer[0] 计算时，异步地将下一个 block 的权重加载到 buffer[1]。计算完成后交换两个 buffer，实现计算与传输的重叠。

### 7.1 权重类对比

### 7.2 推理类对比

| 维度 | WanModel | HunyuanVideo15Model |
|------|----------|---------------------|
| Block 结构 | SelfAttn + CrossAttn + FFN | **DoubleStream + SingleStream** |

HunyuanVideo 的 Transformer 采用 **Double-Stream + Single-Stream** 架构（类似 MMDiT），而 Wan 使用传统的 Self-Attn + Cross-Attn 结构。

这就是三阶段架构的价值——**相同的调用模式，不同的内部实现**。

## 9. sensitive_layer：敏感层的精度保护

```python
# wan/model.py:41-50
self.sensitive_layer = {
    "norm",           # LayerNorm
    "embedding",      # 嵌入层
    "modulation",     # 调制参数
    "time",           # 时间嵌入
    "img_emb.proj.0", # CLIP 投影
    "img_emb.proj.4",
}
```

**什么是 sensitive_layer？**

即使模型整体使用 BF16 推理，某些层（如 LayerNorm、嵌入层）对精度非常敏感。框架维护了一个 `sensitive_layer` 集合，权重加载时这些层会保持更高的精度（`GET_SENSITIVE_DTYPE()`，通常是 float32）。

```python
# base_model.py:295-300 （加载权重时）
with safe_open(file_path, framework="pt", device=device) as f:
    return {
        key: (
            f.get_tensor(key).to(GET_DTYPE())           # 普通层：BF16
            if unified_dtype or all(s not in key for s in sensitive_layer)
            else f.get_tensor(key).to(GET_SENSITIVE_DTYPE())  # 敏感层：FP32
        )
        for key in f.keys()
    }
```

推理代码中也会相应地做精度转换：


## 11. 实现新模型的网络层

如果要适配一个新的 DiT 模型，需要实现：

```
lightx2v/models/networks/my_model/
├── model.py                 # MyModel(BaseTransformerModel)
├── infer/
│   ├── pre_infer.py         # 嵌入逻辑
│   ├── transformer_infer.py # Block 推理逻辑
│   └── post_infer.py        # 输出逻辑
└── weights/
    ├── pre_weights.py        # 前处理权重定义
    ├── transformer_weights.py # Block 权重定义
    └── post_weights.py       # 后处理权重定义（可选）
```

核心步骤：

1. **定义权重类**：在 weights/ 中用 `MM_WEIGHT_REGISTER`、`ATTN_WEIGHT_REGISTER` 等声明权重结构
2. **定义推理类**：在 infer/ 中实现 `infer_block()` 等方法，通过 `weight.apply()` 调用权重
3. **定义 Model 类**：设置 `pre_weight_class`、`transformer_weight_class`、`post_weight_class`，实现 `_init_infer_class()` 和 `_init_infer()`

**通过继承可以大量复用**：如果新模型的 Block 结构和 Wan 类似，可以继承 `WanTransformerInfer` 只覆盖差异部分。

## 13. 思考题

1. **为什么 pre_infer 和 post_infer 不做 Offload？**
   - 提示：比较它们和 transformer_weights 的参数量

   参数量太小了，不是显存大头

   **批改**：正确 ✅。补充一下具体数据：

   以 Wan 2.1 14B 为例：
   - `pre_weight`：patch_embedding + time_embedding + text_embedding + CLIP 投影 ≈ **几十 MB**
   - `transformer_weights`：48 个 Block，每个 Block 有 SelfAttn(Q/K/V/O) + CrossAttn(Q/K/V/O) + FFN ≈ **13GB+**（占总参数的 95% 以上）
   - `post_weight`：Wan 没有 post_weight；HunyuanVideo 的 post_weight 也只有几十 MB

   Offload 的收益 = 节省的显存 - 引入的传输延迟。对于 pre_weight 和 post_weight，节省的显存微乎其微，但每次推理都要做 CPU→GPU 传输，得不偿失。所以只对 transformer_weights（占大头的 Block 权重）做 Offload。

   看代码中的证据（`wan/model.py:165-166`）：
   ```python
   # Offload 时只搬 pre_weight 和 non_block_weights，Block 权重由 offload_manager 管理
   self.pre_weight.to_cuda()
   self.transformer_weights.non_block_weights_to_cuda()  # 只搬 norm/head 等非 Block 权重
   ```

2. **WanModel 的 sensitive_layer 中为什么包含 "norm" 和 "modulation"？**
   - 提示：考虑这些层的数值特征

   不知道

   **参考答案**：

   `norm`（LayerNorm）和 `modulation`（AdaLN 调制参数）对精度敏感的原因不同：

   **LayerNorm**：计算 `(x - mean) / std * γ + β`。`std` 的计算涉及求平方和再开根号，BF16 的精度（尾数只有 7 位，有效精度约 3 位小数）在累加大量元素的平方时容易溢出或精度丢失。用 FP32（尾数 23 位）可以避免这个问题。

   **modulation**：是 AdaLN 的 shift/scale/gate 参数。这些值直接**乘加**到每一层的特征上，误差会在 48 个 Block 中逐层累积。比如每层 scale 有 0.1% 的误差，48 层之后误差放大到 `1.001^48 ≈ 4.9%`。用 FP32 计算可以让每层误差控制在极小范围内。

   简单说：**norm 敏感是因为数值计算本身容易精度丢失，modulation 敏感是因为误差会逐层累积放大**。

3. **如果要给 Wan 模型添加一种新的注意力算子（如 Linear Attention），需要修改哪些文件？**
   - 提示：考虑注册器机制

    添加新的注册器(register_factory.py)，添加新的权重容器

   **批改**：方向部分对 ❌，但有两个关键误解。

   **不需要添加新的注册器**。`ATTN_WEIGHT_REGISTER` 已经存在了，你只需要往里面**注册一个新的实现**。注册器是"字典"，你要做的是往字典里加一条记录，而不是创建一个新字典。

   **也不需要添加新的权重容器**。注意力算子没有可学习权重（它只做 Q、K、V 的 attention 计算），所以不需要新的权重容器。Q/K/V 的线性投影权重已经在 `WanSelfAttention` 中用 `MM_WEIGHT_REGISTER` 定义好了，注意力算子只是决定"拿到 Q、K、V 之后怎么算 attention"。

   **正确的步骤**：

   **只需要改 1 个文件**（新建 1 个文件）：

   ```python
   # 新建 lightx2v/common/ops/attn/linear_attn.py

   from lightx2v.utils.registry_factory import ATTN_WEIGHT_REGISTER
   from .template import AttnWeightTemplate

   @ATTN_WEIGHT_REGISTER("linear_attn")       # ← 注册到已有的注册器
   class LinearAttnWeight(AttnWeightTemplate): # ← 继承已有的模板
       def apply(self, q, k, v, **kwargs):
           # 实现 Linear Attention 计算
           # 比如: φ(Q) @ (φ(K)^T @ V)
           q = torch.nn.functional.elu(q) + 1
           k = torch.nn.functional.elu(k) + 1
           kv = torch.einsum("snd,snf->ndf", k, v)
           return torch.einsum("snd,ndf->snf", q, kv).reshape(q.shape[0], -1)
   ```

   然后确保它被 import（在 `lightx2v/common/ops/attn/__init__.py` 中已有 `from .linear_attn import *` 的模式）。

   使用时只需在配置中指定：
   ```json
   {"self_attn_1_type": "linear_attn"}
   ```

   不需要改 `transformer_infer.py`、`transformer_weights.py` 或任何其他文件。**这就是注册器机制的价值——新增实现不需要改现有代码**。

4. **双缓冲（double buffering）为什么使用 2 个 buffer 而不是 3 个？**
   - 提示：考虑计算与传输的时间比

    为什么需要三个buffer？两个不就能重叠了

   **批改**：结论正确 ✅，但"为什么不需要三个"的推理需要更严谨。关键在于搞清楚**谁是瓶颈**。

   **情况1：计算时间 > 传输时间（DiT 的实际情况）→ 2 个 buffer 够用**

   ```
   GPU 计算:  |======= Block 0 计算 =======|======= Block 1 计算 =======|
   CPU→GPU:   |== Block 1 传输 ==|          |== Block 2 传输 ==|
                                 ↑ 传输早完成               ↑ 传输早完成
                                   等 GPU 算完再 swap          等 GPU 算完再 swap
   ```

   传输比计算快 → 传输**总是在计算结束前就完成了** → GPU 算完 Block 0 时 Block 1 已在 buffer 里 → 直接 swap 继续算 → GPU 零空闲。

   **情况2：传输时间 > 计算时间（假设的极端情况）→ 2 个 buffer 不够**

   ```
   GPU 计算:  |== Block 0 ==|   空闲等待   |== Block 1 ==|   空闲等待   |
   CPU→GPU:   |========= Block 1 传输 =========|========= Block 2 传输 =========|
                                          ↑ GPU 算完了但 Block 1 还没传完！
   ```

   传输比计算慢 → GPU 算完了下一个 Block 还没到 → GPU 空转浪费。

   **那多加 buffer 能解决吗？** 不能。因为 PCIe 传输是串行的——同一时刻只能传一个 Block。假设 load=3s, compute=1s，用 4 个 buffer 预加载：

   ```
   启动：预加载 B0~B3 到 4 个 buffer

   Time 0-1: 计算 B0。所有 buffer 满，没法加载
   Time 1-2: 计算 B1。释放 B0 的 buffer，开始加载 B4（需要 3s）
   Time 2-3: 计算 B2。B4 加载中（1/3）
   Time 3-4: 计算 B3。B4 加载中（2/3）
   Time 4-5: B4 刚好到！计算 B4。开始加载 B5
   Time 5:   B4 算完，但 B5 才加载了 1/3 → GPU 空转 2s
   Time 7-8: B5 到了，计算 B5 → 又空转 2s
   ...
   ```

   预加载只是让开头几个 Block 能连续跑，一旦耗尽就**回到和 2 个 buffer 一样的稳态**：算 1s 等 2s，利用率 1/3。

   **根本原因**：PCIe 传输是串行的，不管有多少 buffer，同一时刻只能传一个 Block。传输带宽是硬瓶颈，buffer 数量无法突破它。

   ```
   瓶颈在传输时（load > compute）：
     稳态 GPU 利用率 = compute_time / load_time = 1/n
     与 buffer 数量无关，加 buffer 只延迟问题爆发但不解决问题
     真正的解法：减少传输量（phase 粒度）或提高带宽

   瓶颈在计算时（compute > load，DiT 实际情况）：
     稳态 GPU 利用率 = 100%
     2 个 buffer 就够
   ```

   **结论：计算慢 = 传输有充足时间在计算窗口内完成 = 2 个 buffer 完美重叠。传输慢时加 buffer 也没用，因为 PCIe 是串行瓶颈。**

   DiT 的实际情况属于情况1：一个 Block 的权重约 300MB，PCIe 4.0 传输约 10-15ms；但一个 Block 的计算涉及数万 token 的多次矩阵乘法 + 注意力，耗时远超传输。所以 2 个 buffer 就足够。

   看代码中 `swap_blocks()` 的实现（`manager.py:79-88`）：
   ```python
   def swap_blocks(self):
       self.cuda_load_stream.synchronize()   # 等传输完成（几乎不阻塞）
       self.compute_stream.synchronize()      # 等计算完成（这才是瓶颈）
       # 交换两个 buffer 的角色
       self.cuda_buffers[0], self.cuda_buffers[1] = self.cuda_buffers[1], self.cuda_buffers[0]
   ```
